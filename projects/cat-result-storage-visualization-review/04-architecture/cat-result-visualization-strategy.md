# CAT 결과 vs 시계열 관측 데이터 역할 차이(시각화 레이어 전략)

**Date:** 2026-03-19

## 1) CAT에서 “최종 결과”와 “시계열 관측 데이터”를 분리해야 하는가?
- **YES.** 분리하지 않으면 아래 문제가 생긴다.
  - `cat-result.json`은 run 단위의 판정/근거(기계 판정)가 필요하고
  - 시계열 저장소는 “관측/디버깅/시간 흐름”이 목적이다.
- 둘의 소비자가 다르다.
  - CAT 최종 판정: 오케스트레이터/게이트/회귀 비교
  - 관측 시계열: 원인 분류/추세 확인

## 2) cat-result.json 같은 판정 결과 vs InfluxDB 같은 시계열 저장소 역할
- `cat-result.json`
  - 목적: PASS/FAIL 단언(권위) + selected_sli 근거
  - 저장 단위: test/job run
  - 스키마: 도구 공통
- InfluxDB
  - 목적: 분포/추세/구간 변화 같은 “시간 흐름”을 관측
  - 저장 단위: metric point(시계열)
  - 스키마: metric 중심(도구/쿼리 중심)

## 3) Evidence를 붙일 때 가장 자연스러운 레이어
- 추천
  - 1차: `cat-result.json` 기반 Evidence 리포트(테이블/요약/판정 히스토리)
  - 2차(필요 시): 실행 중/구간의 관측 지표는 InfluxDB에서 가져와 “원인 분류 페이지”로만 사용
- 이유
  - Evidence는 SQL+마크다운 렌더링이 강하고, `cat-result`는 구조화가 쉽다.
  - 시계열은 과도한 데이터량/스키마 복잡도를 유발하므로 “필요할 때만” 붙이는 게 단순하다.

## 4) 최종적으로 CAT에서 무엇은 파일로 남기고, 무엇은 DB로 넣을까?
- 파일로 남길 것(고정)
  - raw 결과(도구 raw)
  - `cat-result.json`(표준 판정/selected_sli)
- DB로 넣을 것(선택)
  - “실행 중 추세/구간 분포”가 필요한 일부 시계열만
  - 특히 k6에서 already 쉽게 만들 수 있는 metric부터 단계적으로
