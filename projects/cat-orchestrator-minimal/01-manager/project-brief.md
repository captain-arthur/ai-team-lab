# CAT Orchestrator 최소 구조 설계(실무 도입)

**Date:** 2026-03-19  
**Program:** CAT

## 왜 필요한가
- 우리는 이미 `k6`, `clusterloader2(CL2)`, `Ginkgo` 같은 **도구**를 확보했다.
- 그러나 “CAT 시스템”으로 쓰려면, 도구들을 같은 방식으로 실행하고, 같은 방식으로 결과를 저장/표준화하며, 최종 PASS/FAIL을 일관되게 다루는 **오케스트레이션 구조**가 필요하다.

## 이번 설계 범위
- **포함**
  - Job Spec(입력 표준)
  - Runner(도구 선택/실행)
  - Adapter(도구 raw → `cat-result.json` 정규화)
  - Result Store(파일 저장 규약)
  - (옵션) 전체 PASS/FAIL 판단은 “가능한 형태”만 언급(구현 범위 밖)
- **제외**
  - 복잡한 플러그인 시스템/분산 오케스트레이션
  - DB/Grafana 기반 “결과 저장 대체”
  - 도구(k6/CL2/Ginkgo) 내부 수정
  - UI/대시보드/스케줄러

## 성공 기준
다음이 읽는 사람에게 바로 실행 느낌이 나야 한다.
1. `k6` / `CL2` / `Ginkgo`를 **같은 Runner 패턴**으로 실행 가능(도구별 실행 커맨드만 바뀜).
2. 결과가 항상 **동일 포맷**(`cat-result.json`)으로 저장됨.
3. PASS/FAIL의 권위가 명확함
   - `tool exit code`가 최종 판정이며, CAT는 재판정하지 않는다.
4. 설계가 과하지 않고, “CLI로 바로 묶어 실행”되는 수준이다.
