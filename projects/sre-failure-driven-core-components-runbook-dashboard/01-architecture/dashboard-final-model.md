# 최소 지표 기반 운영 대시보드 최종 모델

이 문서는 “설명”이 아니라, 9개 패널이 반드시 같은 방식으로 판정되도록 **최종 모델**을 고정한다.

## 1) 3개 질문 재정의(대시보드의 단 하나의 역할)

### Q1. 지금 안전한가?
정의: **지금 사용자 영향이 허용 가능한가**를 의미한다.
- 판정 단위: 컴포넌트 3개 각각에 대해 safe / warning / fail로 분류
- evidence(판정 근거): 성공/실패 품질과 타임아웃/에러의 “현재 상태”

### Q2. 내일도 안전한가?
정의: **오늘의 상태가 내일 운영 창에 악화 신호를 남기는가**를 의미한다.
- 판정 단위: 컴포넌트 3개 각각에 대해 stable / risk / high로 분류
- evidence(판정 근거): 최근 구간에서의 악화/회복 여부(“추세”)

### Q3. 지금 무엇을 해야 하는가?
정의: Row1/Row2 판정 결과를 기반으로 **다음 5~10분 안에 할 1차 행동**을 고정한다.
- 판정 단위: 컴포넌트별 1개 “action class”(첫 행동 1줄)
- evidence(판정 근거): Q1/Q2에서 만들어진 상태(해석이 아니라 분류)

## 2) 각 질문이 요구하는 판정 기준(고정 규칙)

### Q1 판정(현재)
모든 컴포넌트에 공통으로 적용되는 규칙은 아래 3가지다.
1. 품질이 유지되고 준비성이 정상 => `safe`
2. 품질이 일부 흔들리거나 타임아웃/에러가 소폭 증가 => `warning`
3. 품질 붕괴 또는 실패가 지속/확대되고 준비성이 흔들림 => `fail`

### Q2 판정(내일)
1. 악화 징후가 보이지 않고 회복이 관측 => `stable`
2. 악화 징후가 완만하지만 지속 => `risk`
3. 회복 실패/반복 악화가 명확 => `high`

### Q3 판정(행동)
- Q3는 “추가 metric 해석”을 금지한다.
- Q1/Q2 상태를 입력으로 받아, 컴포넌트별로 **runbook 상세가 아닌 첫 행동 1줄**만 제공한다.

## 3) 왜 이 3개면 충분한가(과도한 관측 금지)
운영 대시보드가 실패하는 이유는 보통 2가지다.
1) metric이 많아서 “해석 시간”이 늘어남  
2) 해석이 끝나도 “다음 행동”이 화면에서 고정되지 않음

이 모델은 metric → 해석을 줄이기 위해, 질문을 판정 단위로 먼저 고정하고(Q1/Q2) Q3에서 행동을 고정한다.
즉, 이 대시보드는 “관측”이 아니라 “판정과 행동”을 위한 최소 화면이다.
