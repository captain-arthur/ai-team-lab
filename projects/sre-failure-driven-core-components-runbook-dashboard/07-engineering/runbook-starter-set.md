# Runbook Starter Set (대시보드 즉시 연결용)

이 문서는 “대시보드 Action(Row 3)”에서 바로 링크할 수 있게, 컴포넌트별로 최소 체크리스트만 제공한다.

공통 원칙
- 읽는 순서: `첫 5분 체크` → `다음 10분 체크` 순서 고정
- 목표: 원인 분류를 먼저 끝내고, 완화 후보를 빠르게 선택
- 상세 원인/설명은 줄이고, “지금 확인할 것”과 “다음 행동”을 중심으로 작성

---

## 1) Runbook: istio-ingressgateway

### runbook_id
`runbook://istio-ingressgateway/failure/fail`  
`runbook://istio-ingressgateway/risk/high`

### 언제 보는가(트리거 상태)
- `fail`: 사용자 관측 실패가 지속되거나 준비성이 흔들리며 영향이 현실로 관측됨
- `risk/high`: 회복이 되지 않는 반복 패턴이 있고 내일 운영 창에도 영향이 예상됨

### 첫 5분 체크(원인 분류 우선)
1. 게이트웨이 “준비성/헬스”가 현재 정상인지 먼저 확인
2. 트래픽이 실제로 게이트웨이에서 upstream으로 넘어가는지 분기
   - 게이트웨이에서 라우팅/연결 실패 유형이 보이면 “게이트웨이 측 문제” 가능성을 우선
   - upstream으로 넘어가는데도 실패면 “타깃 서비스/엔드포인트 문제” 가능성을 우선
3. 특정 도메인/경로만 집중적으로 악화되는지 확인(선택적 편차)

### 다음 10분 체크(조치 후보 선택)
1. 게이트웨이 리소스/프로세스 포화 징후(큐잉/드롭/서킷차단 등)가 있는지 확인
2. upstream 엔드포인트의 상태(ready/헬스)와 라우팅 정책 변경 이력을 교차 확인
3. 타임아웃/라우팅 관련 설정 변경이 최근에 있었는지 확인(최근 변경이 원인 후보)

### 자주 있는 원인 후보(우선순위)
1. 게이트웨이 준비성/리소스 압박
2. upstream 연결 실패(엔드포인트/서비스 상태)
3. 라우팅/인증/TLS 관련 구성 문제
4. 특정 경로에만 편중된 트래픽 패턴 변화

### 완화/조치 후보(바로 선택 가능한 형태)
1. 즉시 대응: upstream 엔드포인트/라우팅 경로를 “안전한 경로”로 우회(가능한 경우)
2. 게이트웨이 측: 스케일/재시작 등 준비성 복구 조치(리소스 압박이면 용량 대응 우선)
3. 설정 측: 최근 라우팅/타임아웃/인증 관련 변경이 있으면 롤백 또는 보수적 설정으로 전환

### escalation 조건
- 5분 내 원인 분류가 불가하거나, 영향이 여러 라우팅 경로로 확산되는 경우
- 완화 후보를 적용해도 회복이 관측되지 않고 실패가 지속되는 경우
- 보안/인증(TLS/권한) 계열 실패가 명확할 때(해당 팀/운영 체계로 즉시 분기)

## 2) Runbook: nodelocaldns

### runbook_id
`runbook://nodelocaldns/failure/fail`  
`runbook://nodelocaldns/risk/high`

### 언제 보는가(트리거 상태)
- `fail`: 노드 로컬 DNS 질의 실패/타임아웃이 확산되고, 이름 해석 기반 서비스 통신이 현실로 깨짐
- `risk/high`: 로컬 DNS 품질이 누적 악화되고 내일 운영 창에서도 영향이 예상됨

### 첫 5분 체크(업스트림 vs 로컬 분리)
1. upstream(coredns)에 도달하는지부터 분리(로컬은 버틸 수 있는지 확인)
2. nodelocaldns 포드/데몬셋의 준비성/재시작 여부 확인
3. “특정 노드군”에서만 문제인지 확인(노드군 편차가 있으면 원인 좁히기)

### 다음 10분 체크(원인 후보 고정)
1. upstream 품질을 다시 확인하고(분리 결과 재검증), 로컬이 원인이라면 리소스/구성 이슈를 점검
2. 최근 네트워크 정책/노드 변경/라우팅 변경이 있는지 교차 확인
3. 캐시 효과 저하가 지속되는지 확인(로컬이 캐시 역할을 못 하고 있는지)

### 자주 있는 원인 후보(우선순위)
1. upstream(coredns) reachability 악화
2. nodelocaldns 데몬셋/포드 준비성 불안정(재시작/리소스 압박)
3. 네트워크 정책/노드 라우팅 문제
4. 로컬 캐시/포워딩 설정 문제 또는 변경 영향

### 완화/조치 후보
1. upstream이 원인일 때: coredns Row로 조사/완화를 우선 연결(상위 원인으로 처리)
2. 로컬이 원인일 때: nodelocaldns 준비성 복구(재시작/스케일/리소스 완화)
3. 노드군 편차가 명확하면: 영향을 받는 노드군 단위로 제한 완화(롤링 접근)

### escalation 조건
- upstream 분리 결과가 명확해지지 않거나, 짧은 시간에 다수 노드로 확산되는 경우
- 준비성 복구 조치 후에도 품질이 회복되지 않는 경우
- 네트워크 정책/보안 관련 변화가 원인 후보로 떠오를 때(해당 운영 체계로 분기)

## 3) Runbook: coredns

### runbook_id
`runbook://coredns/failure/fail`  
`runbook://coredns/risk/high`

### 언제 보는가(트리거 상태)
- `fail`: 클러스터 DNS 에러/지연이 지속되고 준비성이 불안정하며 서비스 영향이 현실로 관측됨
- `risk/high`: 큐잉/리소스 압박 또는 헬스 플랩 등 내일 전이 위험이 명확

### 첫 5분 체크(원인 분류의 최소 세트)
1. coredns 헬스/준비성 상태가 안정적인지 확인
2. forward/upstream 품질 이슈(업스트림 도달성/응답 품질) 여부를 먼저 분리
3. 구성 변경(설정 로드/플러그인 변경) 또는 최근 재시작 이력과 동시성 확인

### 다음 10분 체크(완화 선택)
1. 리소스/큐잉/워커 소진 등 포화 징후가 있는지 확인
2. 로그 기반으로 “에러 유형”을 분류하고, 설정/업스트림 중 어디가 더 가까운지 결정
3. nodelocaldns의 upstream으로서 역할이 깨진 패턴과 교차 확인

### 자주 있는 원인 후보(우선순위)
1. upstream forward 도달성/품질 문제
2. 리소스 압박으로 인한 처리 지연/큐잉
3. 설정/구성 변경 영향(롤아웃 직후 패턴)
4. 헬스 플랩 또는 플러그인 체인 실패

### 완화/조치 후보
1. upstream 품질이 원인일 때: forward 대상/네트워크 경로 우선 복구
2. 리소스가 원인일 때: 스케일 또는 리소스/설정 보수적 조정으로 즉시 완화
3. 설정 변경이 원인 후보일 때: 롤백을 우선 고려(가장 빠른 안정화)

### escalation 조건
- 10분 내 원인 분류가 되지 않거나, 헬스 플랩이 반복되는 경우
- nodelocaldns까지 연쇄 영향이 확실해지는 경우(클러스터 전파)
- 설정/구성 변경이 동시성으로 의심되면 해당 변경 소유 팀으로 즉시 분기
