# 운영 방법론: Failure / Confirmation / Action

이 문서는 “대시보드의 행위(판정) → 행동(runbook)”이 자연스럽게 이어지도록 최종 방법론을 고정한다.

## 1) 운영 대시보드가 따라야 할 최종 방법론
채택 모델: **Failure / Confirmation / Action**

### Failure(지금 안전한가?)
- 의미: 사용자가 체감하는 실패 증거가 있는가(또는 품질이 무너지고 있는가)
- 출력: 컴포넌트별 상태( `safe / watch / fail` )

### Confirmation(내일도 안전한가?)
- 의미: 오늘의 실패/위험이 내일 운영 창에서 악화될 가능성이 있는가
- 출력: 컴포넌트별 추세 상태( `stable / risk / high` )

### Action(지금 무엇을 해야 하는가?)
- 의미: 위 상태에서 운영자가 다음 5~10분 내 수행할 행동이 무엇인가
- 출력: runbook_id + 첫 5분 체크/다음 10분 체크 연결

## 2) 각 단계의 의미(대시보드 관점)

### 지금 안전한가( Failure Row )
- 역할: “오늘 당장 영향이 있는가?”를 가장 빠르게 분류한다.
- 판정 단위: 컴포넌트(3개)
- 핵심: 실패 증거가 없으면 safe.

### 내일도 안전한가( Confirmation Row )
- 역할: 지금은 안전해 보여도, 악화 조짐이 있는지 승격한다.
- 핵심: 추세 기반 승격만 하고, 상세 원인 진단은 Action에서 한다.

### 지금 무엇을 해야 하는가( Action Row )
- 역할: 상태에 따라 runbook을 열고, 첫 행동을 고정한다.
- 핵심: Action은 “찾기”가 아니라 “다음 행동”이 되어야 한다.

## 3) 장애 체인(압박 → 부담 → 장애)의 위치

채택: **보조 설명 모델로만 제한적 사용**

- 메인 모델이 아니어야 한다.
- 왜냐하면 chain은 설명은 되지만, 운영자가 다음 행동을 5분 내 선택하기 어렵기 때문이다.

따라서 대시보드에서는:
- Failure Row: 품질/성공 실패를 직접 판정(메인)
- Confirmation Row: 추세 승격(보조)
- Action Row: 원인 분류 후 실행(메인)

압박→부담→장애는 “Confirmation Row에서 risk를 만들 때” 정도로만 참고한다.
대시보드 패널에는 자세한 chain을 길게 두지 않는다.

## 4) runbook 연결 모델(최종 추천)
가장 실용적인 연결은 패널 상태 기반이다.

- 패널 상태 → next action → runbook_id/link
- Row 상태를 그대로 “action class”로 매핑해, 컴포넌트별 runbook으로 연결

이 방식은 운영자가 화면에서 “상태”를 보는 즉시 runbook으로 이동하게 해준다.
