# 우선 신호 + threshold 인벤토리(판정에 필요한 최소만)

주의: 아래 목록은 “metric 나열”이 아니라, 각 컴포넌트의 운영 판정에 직접 쓰이는 **signal 단위**와 그 signal이 Row1/Row2에 어떤 상태를 만들지의 초안이다.

## istio-ingressgateway
### Signal TOP
1. 사용자 관측 실패(성공률/오류/타임아웃 패턴)
   - 목적: Row1의 fail/w atch를 결정(지금 안전/위험)
2. 사용자 관측 지연(특히 p95 수준의 지속/회복)
   - 목적: Row1 safe/watch 경계와 Row2 risk 승격
3. 게이트웨이 준비성(ready/health 안정성)
   - 목적: fail의 확증 판정 근거(게이트웨이 자체 문제 분리)
4. 업스트림 연결/라우팅 실패의 유형(게이트웨이에서 분류 가능한 레벨)
   - 목적: Row3에서 runbook 첫 행동(원인 분류)으로 연결
5. 완화/서킷차단/드롭 등 전이 징후(있다면)
   - 목적: “내일도 안전한가”에서 risk-high로 승격하는 보강 판정 근거

### threshold 초안(상태 판정 규칙 예시)
- Row1(오늘)
  - fail: 사용자 관측 실패 또는 타임아웃 증가가 일정 창(예: 연속 2개 윈도우)에서 지속되고 준비성이 불안정
  - watch: 실패는 크지 않지만 지연 p95가 기준치 근접/간헐 초과 또는 준비성의 간헐 흔들림
  - safe: 성공 유지 + 지연/실패가 기준 이내 + 준비성 안정
- Row2(내일)
  - risk: 실패/지연이 “회복되지 않고” 완만한 상승 추세
  - high: 회복 실패 또는 반복 패턴이 확실(특정 도메인/경로 편차 포함)

## nodelocaldns
### Signal TOP
1. 로컬 DNS 질의 성공/실패 비율(시간당 실패 증거)
   - 목적: Row1 fail/watch 판정
2. DNS 질의 타임아웃/응답 지연(특히 노드 편차)
   - 목적: Row1에서 watch 확정, Row2에서 risk-high 승격
3. upstream(coredns) 도달성 신호(로컬에서 관측 가능한 수준)
   - 목적: Row3에서 “upstream vs 로컬” 1차 분기
4. 로컬 DNS 포드/데몬셋 안정성(재시작/준비성 플랩)
   - 목적: fail의 확증 또는 로컬 원인 분류 판정 근거
5. 캐시/포워딩 품질의 저하 경향(캐시 hit 감소/포워딩 지연 등)
   - 목적: Row2 risk를 만드는 누적 증거

### threshold 초안
- Row1(오늘)
  - fail: 성공률 하락과 타임아웃 증가가 확산(여러 노드)되며 지속
  - watch: 일부 노드에서만 실패/지연이 커지거나 간헐적으로 타임아웃 증가
  - safe: 성공률/지연 안정 + 로컬 준비성 유지
- Row2(내일)
  - risk: 노드 편차가 커지며 업스트림 품질이 떨어지는 추세
  - high: 실패가 다수 노드로 확대되고 회복이 느림

## coredns
### Signal TOP
1. DNS 에러 계열의 누적(타임아웃/SERVFAIL 유사/실패 확률)
   - 목적: Row1 fail 확정 및 Row2 악화 추세 판정
2. 응답 지연의 지속(특히 p95/p99 수준)
   - 목적: Row1 경계와 Row2 risk-high 승격
3. 헬스/준비성 플랩 또는 재시작 빈도
   - 목적: 운영적 “불안정” 판정 근거
4. forward/upstream 품질(코어DNS 관점)
   - 목적: 원인 분류 및 runbook 첫 행동 결정
5. 처리 포화 징후(큐잉/워커 소진/리소스 압박)
   - 목적: 내일 위험(risk-high) 승격에서 핵심

### threshold 초안
- Row1(오늘)
  - fail: 에러율 급상승 또는 지연이 지속되며 준비성 불안정
  - watch: 에러 또는 지연이 회복되지 않고 간헐 증가
  - safe: 헬스 안정 + 에러/지연 기준 이내
- Row2(내일)
  - risk: 에러/지연이 완만히 증가하고 회복 실패 징후가 존재
  - high: 준비성 플랩/재시작 반복 + 누적 지연
