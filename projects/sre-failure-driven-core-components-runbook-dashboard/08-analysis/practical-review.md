# 실무 타당성 리뷰(냉정 점검)

## 1) 이 구조가 실무적으로 유효한가?
유효하다고 판단한다. 이유는 row/panel이 “관측”을 끝내지 않고 곧바로 상태 판정과 runbook 링크를 연결하도록 강제했기 때문이다.

핵심은 다음이다.
- Row 1/2는 상태 판정(안전/위험, 안정/리스크)을 만들고
- Row 3는 상태에 따른 next action과 runbook_id를 고정한다

이 구조는 운영자가 “그래서 지금 뭘 하지?”를 즉시 해결하게 만든다.

## 2) 패널 → action → runbook 흐름이 자연스러운가?
자연스럽다.  
이유는 Action(Row 3)의 역할이 “원인 분석”이 아니라 “첫 확인과 다음 체크리스트로 진입”이기 때문이다.

즉,
- panel은 1~2단계의 첫 행동을 요약하고
- runbook은 실제 체크리스트(첫 5분/다음 10분/원인 후보/완화 후보)를 제공한다

이 역할 분리가 “대시보드가 runbook까지 다 흡수하려는 과설계”를 막는다.

## 3) 패널 수가 과하지 않은가?
과하지 않다.  
대상 3개 컴포넌트를 위해 최소 패널(Confidence 3 + Risk 3 + Action 3 = 총 9)만을 요구했다.

다만 실제 구현에서는 아래 규칙을 지켜야 한다.
- 동일 지표를 Row 1/2에 반복해서 보여주지 말 것
- 각 패널은 “질문 1개에 대한 답(상태/행동)”을 가져야 함

## 4) 운영자가 정말 5~10분 안에 판단 가능한가?
전제가 있다.
- 상태 판정 기준이 운영팀의 실제 경험과 맞게 튜닝되어야 한다(초안 기준은 문서에 존재)

그 외에는 가능한 설계다.
- Row 3 Action이 “첫 확인 항목 1~3개”로 runbook 진입을 강제하므로,
  운영자는 탐색 시간을 줄이고 행동을 시작한다.

## 5) 철학이 실제 설계에 반영되었는가?
반영됐다.
- “failure-driven”은 metric 나열이 아니라 실패/위험 판정과 runbook 링크로 번역됐다.
- TDD/Kent Beck의 번역도 “작은 판정 단위(컴포넌트 단위) + 실패 증거를 먼저 드러냄 + 행동 가능한 피드백 루프”로 대시보드 구조에 들어갔다.

즉 장식이 아니라, row/panel이 그 제약을 따르도록 설계되었다.

## 6) 과감히 버려야 할 부분은 무엇인가?
다음은 버려야 한다.
- “압박 → 부담 → 장애”를 메인 모델로 계속 그리는 것
  - 이 문서에서는 이를 보조 설명으로만 제한했고, action flow를 흐리지 않도록 했다.
- 실패와 무관한 장식용 metric을 추가하는 것
  - 무엇을 해야 하는지에 기여하지 못하면 제거한다.
- runbook을 너무 길게 쓰는 것
  - runbook starter는 “첫 5분/다음 10분” 중심으로 고정해야 한다.

## 결론
이 설계는 운영 현실(짧은 골든 타임, 빠른 분류, runbook 진입)을 전제로 하며, 과설계 없이 action 연결을 강제하므로 실무 타당성이 높다.
