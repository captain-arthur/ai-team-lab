# 장애 주도 대시보드 + 대시보드 주도 액션(운영 번역)

이 문서는 “멋있는 비유”가 아니라, 실제 대시보드 설계 원리로 번역한다.

## 1. 장애 주도 대시보드란 무엇인가?

### metric-driven 대시보드와의 차이
- metric-driven
  - 지표를 많이 보여주고, 운영자가 직접 “무엇이 문제인지” 해석한다.
  - 문제 인식 시간과 원인 분류 시간이 늘어날 수 있다.
- 장애(failure) 주도
  - 사용자가 체감하는 실패를 먼저 판정한다.
  - 즉, “지금 안전/위험”을 먼저 분류하고, 그 분류가 다음 행동으로 이어진다.

### incident-driven / failure-oriented의 장점과 한계
- 장점
  - 실패 판정이 명확해지고, runbook 진입이 자연스럽다.
  - 운영자가 보는 화면이 “검색”이 아니라 “행동”으로 바뀐다.
- 한계
  - 실패 판정 기준(증거)이 모호하면 metric-driven처럼 다시 돌아간다.
  - 따라서 이 문서는 실패 판정 기준과 판정 단위를 강제한다.

## 2. 대시보드 주도 액션이란 무엇인가?

대시보드 주도 액션의 의미는 “대시보드가 사람을 대신한다”가 아니다.  
운영자가 **다음 5분** 안에 선택해야 하는 행동이 명확해지는 상태를 만든다.

### 대시보드는 어디까지 해야 하는가
- 판정
  - 지금 안전한가 / 위험한가를 component 단위로 분류한다.
- 추천(최소 행동)
  - 첫 확인 항목 1~3개 수준으로 고정한다.
- 연결
  - runbook_id로 연결해서 “다음 단계의 체크리스트”를 바로 연다.

### runbook과 어디서 만나는가
- “Action(Row 3)”에서 만난다.
  - 패널 상태(예: safe/watch/fail, stable/risk/high)에 따라 runbook 링크가 바뀐다.
  - runbook은 사람의 다음 행동을 구조화한다.

## 3. 운영 판단을 주도하는 원리는 무엇인가(후보 비교)

후보들은 서로 겹칠 수 있지만, 최종적으로는 하나의 주도 원리를 고정해야 한다.

## 비교 기준
- 주도 원리가 “판정 단위”를 만들 수 있는가
- 그 판정 단위가 “다음 행동”으로 바로 이어지는가
- 모호함이 생기면 어디서 다시 돌아오는가(metric 해석으로 돌아오는지)

### 후보별 평가
- confidence-driven
  - 장점: 안전/위험을 정면으로 판정 가능
  - 한계: confidence를 계산하는 방식이 모호해지면 metric-driven으로 복귀
- decision-driven
  - 장점: “이 화면에서 운영자가 결정해야 하는 것”이 명확해짐
  - 한계: decision이 무엇인지 정하지 않으면 비어 있음
- action-driven
  - 장점: runbook 연결이 자연스럽다
  - 한계: action이 앞서면 근거(판정) 구조가 약해질 수 있다
- failure-driven
  - 장점: 사용자가 체감하는 실패를 증거로 삼기 쉬움
  - 한계: failure를 “어떤 의미의 failure로” 정의할지가 필요

## 최종 채택 원리
채택: **failure-driven + decision-driven + action-driven의 결합**

정리:
- failure-driven이 “무엇을 판정할지”를 주도한다(실패 중심 증거)
- decision-driven이 “어떤 단위로 판정할지”를 주도한다(component별 상태)
- action-driven이 “판정이 곧 다음 행동”이 되게 만든다(runbook 연결)

## 4. TDD / Kent Beck 철학을 운영 대시보드로 번역(실제 설계 제약)
Kent Beck/TDD를 운영 설계 원리로 번역하면 다음 제약이 필요하다.

- 빠른 피드백
  - 운영자가 5~10분 내 “상태 분류 + 다음 행동”을 끝내야 한다.
  - 그러려면 row/panel 구조가 짧아야 한다.
- 작은 판정 단위
  - 전체 시스템이 아니라, 컴포넌트 단위로 작은 테스트처럼 판정한다.
- 실패를 먼저 드러냄
- 안전은 기본값이 아니라, failure 판정 근거가 없을 때만 safe가 된다.
- 다음 행동을 쉽게 만듦
  - Action 패널 상태가 곧 runbook_id로 연결되어야 한다.

최종 번역:
대시보드는 “테스트 케이스”처럼 failure 판정 기준을 작게 만들고, 상태에 따라 “바로 다음 체크리스트”를 제공해야 한다.
