# 기존 레퍼런스 패턴(결과 정규화 관점만)

## Kubernetes test-infra / TestGrid
- “실행 결과”를 사람들이 비교·추적할 수 있게 만들기 위해, 테스트 결과는 보통 **표준 형식(예: JUnit XML) + run 디렉터리 구조**로 남긴다.
- 정규화의 포인트는 도구별 내부 구조를 그대로 두지 않고,
  - 공통으로 읽는 필드(테스트명/성공여부/아티팩트 경로)
  - 테스트별 메트릭(있다면)
  을 **같은 위치/형태**로 맞추는 것이다.

## ClusterLoader2 결과 처리 패턴(우리 CAT 맥락)
- CL2는 “측정 결과”를 결과 디렉터리 아래에 여러 파일로 남긴다(요약 JSON, JUnit, 로그 등).
- CAT에서 필요한 정규화는:
  - CL2가 남긴 raw 중 CAT가 요구하는 **selected SLI만 추출**
  - 그 값을 도구 공통 결과 스키마로 **key/value로 정규화**
  - PASS/FAIL은 “정의된 비교 규칙”과 “필수 gating”에 따라 최종 기록
  이 흐름에 맞춘다.

## Google/신뢰 가능한 테스트 시스템의 공통 교훈(정규화 관점)
- 테스트 결과는 자동 수집/누적/회귀 비교가 가능한 단위로 남겨야 한다.
- 그래서 “결과”는 보통
  - 실행 식별자(run-id, test-name)
  - 판정(통과/실패)
  - 근거(어떤 SLI/어떤 규칙)
  를 같이 묶는 형태로 정규화한다.

## Prometheus / OpenTelemetry 데이터 모델에서 가져올 점(단, 전부 흡수하지 않기)
- 시계열에서 흔히 쓰는 구조는 “metric name + labels + value + timestamp”다.
- CAT 정규화에서도 영향을 받을 부분은 다음뿐이다.
  - selected SLI를 동일한 이름 규칙으로 유지
  - (optional) tags/labels로 그룹화 가능한 여지 제공
- 하지만 CAT의 최종 판정 결과는 “시계열을 위한 모델”이 아니라 “run 판정을 위한 요약”이므로, 시계열 DB/텔레메트리 모델 전체를 가져오지는 않는다.

## Evidence 입력/시각화 관점
- Evidence는 SQL+마크다운 렌더링을 위해, 데이터 소스가 쿼리 가능 형태(예: Parquet/CSV 형태)로 준비돼야 한다.
- 따라서 CAT 정규화의 목표는:
  - Evidence가 읽기 쉬운 “표준화된 요약 데이터(메트릭/판정)”를 제공
  - raw는 증거로 유지하되, Evidence 기본 차트는 정규화 결과에서 출발
  이다.
