# 정규화 옵션 비교(JSON/CSV/Influx/Prom 스타일)

이번 비교는 “CAT 최종 결과를 Evidence/비교/누적에 쓰기 위한 정규화 포맷” 관점이다.
시계열(분포 변화)은 별도 저장소로 분리할 수 있다.

## 옵션 A. JSON 표준화(권장 후보)
### 구조
- 각 Job run마다 `cat-result.json` 1개를 저장
- `metrics[]` 배열로 selected SLI만 값/단위/태그를 담는다

### 평가
- 구조 단순성: 높음(파일 1개로 끝)
- 도구 독립성: 높음(k6/CL2/Ginkgo 모두 dict→json 변환으로 수용)
- 시각화 용이성: 중간
  - Evidence가 JSON을 직접 읽는지 여부에 따라 달라짐
  - 하지만 “추가로 CSV 1개”만 생성하면 바로 해결 가능
- 비교/누적 용이성: 높음(파일 단위 diff/집계 쉬움)
- 운영 난이도: 낮음(서버/DB 불필요)
- k6 편향 위험: 낮음(HTTP metric 외에도 key/value 확장으로 흡수)

## 옵션 B. CSV 표준화
### 구조
- run 단위로 1행(또는 run+metric 단위 2개 테이블)로 저장
- Evidence가 즉시 테이블을 만들 수 있음

### 평가
- 구조 단순성: 높음(가독성/조회 쉬움)
- 도구 독립성: 중간~높음(필드 고정이 필요)
- 시각화 용이성: 높음(Evidence SQL에 바로 쓰기 쉬움)
- 비교/누적 용이성: 높음(append-only 테이블로 운영 가능)
- 운영 난이도: 낮음
  - 다만 metric 개수가 늘어나면 열/스키마 관리가 필요
- k6 편향 위험: 중간
  - k6 metric 이름을 열/키로 박아버리면 다른 도구 적합성이 떨어질 수 있음

## 옵션 C. InfluxDB 중심
### 구조
- tool metrics을 시계열 포인트로 저장하고 Evidence는 Influx 기반 시각화
- pass/fail도 “특정 measurement/필드”로 넣을 수 있으나, 이는 선택

### 평가
- 구조 단순성: 낮음(스키마/태그 설계 필요)
- 도구 독립성: 중간(모든 도구를 동일한 time series 모델로 맞춰야 함)
- 시각화 용이성: 높음(시계열에는 강함)
- 비교/누적 용이성: 낮음~중간
  - “run 단위 요약” 비교를 위해 필터/쿼리 규칙이 복잡해질 수 있음
- 운영 난이도: 높음(DB 운영/카디널리티/권한)
- k6 편향 위험: 높음(k6는 Influx output이 자연스럽지만 CL2/Ginkgo는 억지 정합 위험)

## 옵션 D. Prometheus/OpenMetrics 스타일
### 구조
- metric name + labels + value 형태로 저장/노출
- Evidence는 이걸 직접 읽거나 export 변환을 통해 사용

### 평가
- 구조 단순성: 중간(스키마는 정해져 있으나 run 단위 요약 표현이 애매할 수 있음)
- 도구 독립성: 중간~높음(라벨로 도구/시나리오를 담을 수 있음)
- 시각화 용이성: 중간(시계열은 좋지만 “run 단위 PASS/FAIL”에 덜 직관적)
- 비교/누적 용이성: 중간
  - run 단위 집계를 위한 추가 쿼리/규칙이 필요
- 운영 난이도: 중간(시계열 DB/쿼리 계층이 추가됨)
- k6 편향 위험: 중간(도구별 metric 명명/라벨 설계가 관건)

## 최종 추천(또는 단계적 전략)
- **1단계(최소/권장)**: 옵션 A(JSON)로 `cat-result.json` 정규화(단일 소스 오브 트루스)
- **2단계(필요 시, 최소 추가)**: 옵션 B(CSV)는 Evidence 호환을 위해 “선택적으로” `cat-result-table.csv`만 생성
  - 이 csv는 JSON의 파생물(ETL 금지 수준으로 단순 생성)
