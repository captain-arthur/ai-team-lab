# PodStartupLatency 변동성 분석

**Project:** sample-cl2-sli-slo-analysis  
**Input:** slo-validation-analysis.md, 반복 실험 결과(4 runs), experiment-notes.md

**목적:** PodStartupLatency·StatelessPodStartupLatency가 run 간에 **왜 크게 변하는지** 원인을 정리하고, 추가 실험 제안 및 **레이턴시 SLO 가설 수정 방향**을 제안한다. 최종 프로덕션 SLO 수치는 정하지 않는다.

---

## 1. 관측된 변동성 요약

- **PodStartupLatency (pod_startup, P99):** 64.1 s ~ 122.3 s (약 1.9배 차이).
- **StatelessPodStartupLatency (pod_startup, P99):** 10.7 s ~ 69.9 s (약 6.5배 차이).
- **환경:** kind 3노드, 동일 시나리오·오버라이드, Prometheus 비활성.
- **안정성 지표:** ClusterOOMsTracker failures·SystemPodMetrics restartCount는 4 run 모두 0으로 **안정**. 이들은 이미 검증에서 reasonable한 기준점으로 확인됨.

---

## 2. 변동 원인 후보 및 분류

가능한 원인을 나열하고, **기존 실험 결과와 환경 정보만**으로 **likely / possible / unclear**로 분류했다.

### 2.1 Image pull / cold start

| 항목 | 설명 |
|------|------|
| **의미** | 첫 run 또는 이미지가 노드에 없을 때 pull이 발생하고, 이후 run에서는 캐시된 이미지로 기동해 레이턴시가 줄어들 수 있음. |
| **관측과의 관계** | 로그에 "No images specified. Skipping image preloading"이 있으므로 **이미지 사전 풀 없음**. run 간에 어떤 노드에 어떤 이미지가 캐시되어 있는지는 제어되지 않음. |
| **분류** | **Likely.** 이미지 풀 유무·노드별 캐시 상태가 run마다 달라질 수 있어, P99 변동의 한 요인으로 보는 것이 타당함. |

### 2.2 Cluster warm-up

| 항목 | 설명 |
|------|------|
| **의미** | 클러스터 기동 직후 또는 유휴 후 첫 부하일 때 API·스케줄러·kubelet이 “따뜻해지기” 전이라 지연이 크고, 연속 run에서는 상대적으로 짧아질 수 있음. |
| **관측과의 관계** | 4 run 순서가 20260315-181750 → slo-validation-1 → 2 → 3. **가장 빠른 P99는 181750(64.1 s, Stateless 10.7 s)** 이고, validation-2가 가장 느림(122.3 s, 69.9 s). “첫 run이 가장 느리다”는 패턴은 없음. |
| **분류** | **Possible.** warm-up 효과가 있을 수는 있으나, 현재 run 순서만으로는 “첫 run vs 이후 run”을 구분할 근거가 부족함. **Unclear**로 두어도 됨. |

### 2.3 Kind 노드 리소스 압박

| 항목 | 설명 |
|------|------|
| **의미** | kind 노드는 호스트 자원을 나눠 쓰므로, 동시에 많은 파드가 스케줄·기동될 때 CPU/메모리 경합으로 스케줄링·컨테이너 기동이 지연될 수 있음. |
| **관측과의 관계** | P99가 64 s ~ 122 s로 크게 흔들리고, Stateless만으로도 10.7 s ~ 69.9 s로 변동이 큼. 리소스 모니터링은 하지 않았음. |
| **분류** | **Likely.** kind·소규모 노드에서 create phase 부하가 run마다 다르게 겹치면 리소스 압박으로 지연이 늘어나는 것은 충분히 가능함. |

### 2.4 Scheduling delay

| 항목 | 설명 |
|------|------|
| **의미** | 스케줄러가 파드를 노드에 할당하기까지 걸리는 시간. 대기열·스케줄러 부하·노드 리소스에 따라 변동. |
| **관측과의 관계** | PodStartupLatency 하위 구간에 `schedule_to_watch` 등이 있으나, run별로 분해하지는 않음. Create phase에서 많은 파드를 한꺼번에 생성하면 스케줄링 지연이 P99에 포함됨. |
| **분류** | **Likely.** “파드 생성 → Ready” 전체 레이턴시에 스케줄링 구간이 포함되며, 부하·리소스에 따라 run 간 차이가 날 수 있음. |

### 2.5 Storage / network

| 항목 | 설명 |
|------|------|
| **의미** | 볼륨 마운트·네트워크 정책·CNI 등으로 인한 지연. kind는 보통 단일 노드처럼 동작하는 네트워크를 쓰며, 로컬 디스크 성능도 호스트에 의존. |
| **관측과의 관계** | Stateful vs Stateless 차이를 보면 스토리지 관련 지연 가능성은 있으나, **어느 run에서 스토리지/네트워크가 병목이었는지**에 대한 데이터는 없음. |
| **분류** | **Possible.** 원인 후보이지만, 현재 데이터만으로 비중을 판단하기 어렵다. |

### 2.6 Control-plane 병목

| 항목 | 설명 |
|------|------|
| **의미** | API server·스케줄러·컨트롤러 매니저가 과부하일 때 요청 처리·스케줄링이 지연됨. kind는 컨트롤 플레인도 같은 노드에서 돌아가는 경우가 많아 리소스 경합 가능. |
| **관측과의 관계** | 실험에서 SchedulingMetrics는 kind 스케줄러 엔드포인트 접근 실패로 수집되지 않음. 컨트롤 플레인 메트릭(CPU/메모리·요청 지연)은 수집하지 않음. |
| **분류** | **Possible.** 병목 가능성은 있으나, **데이터 부재**로 likely까지 올리기 어렵다. |

### 2.7 First-run vs later-run

| 항목 | 설명 |
|------|------|
| **의미** | “첫 run만 이미지 풀·cold start로 느리고, 이후 run은 캐시·warm-up으로 빠르다”는 패턴. |
| **관측과의 관계** | 181750이 **가장 빠른** P99를 보임. 즉 “첫 run이 가장 느리다”가 아님. run 순서와 레이턴시 순서가 일치하지 않음. |
| **분류** | **Unclear.** 현재 run 순서만으로는 first-run 효과를 주장하거나 배제하기 어렵다. 이미지 캐시·호스트 상태가 날짜/시간에 따라 달랐을 수 있음. |

### 2.8 요약 표

| 원인 | 분류 | 비고 |
|------|------|------|
| Image pull / cold start | **Likely** | 이미지 preload 없음; 노드별 캐시 상태 run마다 상이할 수 있음. |
| Cluster warm-up | Possible / Unclear | 첫 run이 가장 느리다는 패턴 없음. |
| Kind 노드 리소스 압박 | **Likely** | 소규모·공유 자원 환경에서 부하 변동과 일치. |
| Scheduling delay | **Likely** | 전체 startup 레이턴시에 포함; 부하에 따라 변동 가능. |
| Storage / network | Possible | 데이터 부족으로 비중 불명. |
| Control-plane 병목 | Possible | 메트릭 미수집으로 확인 불가. |
| First-run vs later-run | **Unclear** | 관측 패턴이 “첫 run 느림”과 반대. |

## 3. 원인 분리에 도움이 되는 추가 실험

아래 실험으로 위 원인들을 조금씩 구분할 수 있다.

### 3.1 Warm-up run 후 측정

- **방법:** 동일 시나리오를 **한 번 “웜업”으로만** 실행한 뒤, **그 다음 run**의 PodStartupLatency만 기록.
- **의미:** cluster warm-up·이미지 캐시를 어느 정도 채운 상태에서의 “안정 구간” 레이턴시를 볼 수 있음.
- **기대:** warm-up이 주요 원인이라면, 웜업 후 run의 P99가 웜업 없이 연속 실행한 run보다 낮고 안정적일 수 있음.

### 3.2 동일 이미지 캐시 상태에서 반복

- **방법:** **이미지 preload**를 시나리오 또는 CL2 옵션으로 켜서, 모든 노드에 사용 이미지를 미리 풀한 뒤, **동일 config로 3~5회 연속** 실행.
- **의미:** image pull / cold start 효과를 줄인 상태에서의 변동성만 관측.
- **기대:** 캐시 후에도 P99가 run 간에 크게 흔들리면, 리소스·스케줄링·컨트롤 플레인 등 **다른 요인**이 더 크다는 방향으로 해석 가능.

### 3.3 시나리오 규모 축소

- **방법:** create phase 파드 수(또는 객체 수)를 **줄인** 시나리오로 동일한 run 횟수 반복.
- **의미:** 리소스 압박·스케줄링 대기열을 완화해, “부하 크기”와 P99 변동의 상관을 봄.
- **기대:** 규모를 줄였을 때 P99가 낮고 변동이 작다면, **리소스/스케줄링 부하**가 원인일 가능성이 커짐.

### 3.4 노드 리소스 확인

- **방법:** create phase **직전·직후**에 `kubectl top nodes` 또는 노드 메트릭(CPU/메모리 사용률)을 수집.
- **의미:** kind 노드가 리소스 포화에 가까웠는지 여부를 run별로 비교.
- **기대:** P99가 큰 run에서 노드 사용률이 높다면, **리소스 압박**이 likely로 올라감.

### 3.5 컨트롤 플레인 리소스·지연 모니터링

- **방법:** Prometheus 등으로 API server·scheduler 요청 레이턴시·CPU/메모리를 수집하거나, 테스트 구간 전후에 API 응답 시간을 샘플링.
- **의미:** control-plane 병목 여부를 직접 확인.
- **기대:** API/스케줄러 지연이 run별로 다르고, PodStartupLatency P99와 상관이 있으면 **control-plane**이 원인 후보로 부각됨.

### 3.6 실험 우선순위 제안

1. **이미지 preload + 연속 3~5 run:** 구현이 쉽고, image pull/cold start를 가장 직접적으로 줄일 수 있음.  
2. **Warm-up 1회 + 측정 1~2 run:** 웜업 효과 유무를 빠르게 확인.  
3. **시나리오 규모 축소:** 오버라이드·시나리오 수정이 필요하지만, 부하와 변동성 관계를 보는 데 유용.  
4. **노드/컨트롤 플레인 메트릭:** 인프라·도구가 준비되면 원인 분리에 큰 도움.

## 4. PodStartupLatency 관련 SLO 가설 수정 방향

현재 변동성과 원인 분석을 바탕으로, **임계값을 구체적으로 정하지 않고** “어떻게 가설을 바꿀지”만 제안한다.

### 4.1 Local/kind vs 실제 클러스터 수용 구분

- **제안:** **로컬(kind) 수용**과 **실제 클러스터(또는 스테이징) 수용**에서 **서로 다른 SLO를 둘 것**을 권장.
- **이유:** kind는 리소스 공유·이미지 풀·네트워크 스택이 실제 클러스터와 다르고, 현재 관측된 P99(64 s ~ 122 s)는 **이 로컬 환경의 특성**을 반영한 것으로 보는 것이 타당함. 따라서 “kind에서의 수용”과 “실제 클러스터에서의 수용”을 같은 숫자로 묶지 않고, **환경별로 별도 정당화**하는 편이 낫다.

### 4.2 로컬 실험용 임시 베이스라인 SLO

- **제안:** kind·동일 시나리오에서 **현재처럼 반복 실험으로 얻은 분포(예: P99 중앙값·상위 백분위)**를 기준으로, **“로컬 실험용 임시 SLO”**를 둔다.
- **예시 (수치 고정이 아님):** “동일 config 4 run의 P99 중앙값의 1.2배 이하” 또는 “4 run 중 3 run이 통과하는 임계값” 같은 **데이터 기반 임시 기준**.
- **목적:** 프로덕션 SLO가 아닌, **로컬에서 “이번 환경이 이전과 크게 나빠지지 않았는지”**를 보는 용도. 레이턴시 원인이 더 정리되면 임시 기준도 조정.

### 4.3 레이턴시 SLO를 “provisional”로 두기

- **제안:** **더 제어된 환경**(이미지 preload, 리소스 모니터링, 또는 실제 클러스터)에서 실험이 쌓일 때까지, PodStartupLatency·StatelessPodStartupLatency 관련 SLO는 **provisional(가칭)** 으로 두고, “검증 중”으로 명시한다.
- **이유:** 현재는 원인 분리가 불완전하고, kind에서의 변동이 크기 때문에, **최종 프로덕션 SLO**로 고정하기에는 이르다. provisional로 두고, 추가 실험과 환경이 갖춰지면 “확정 SLO”로 승격하는 절차를 두는 것이 안전하다.

### 4.4 OOM·재시작 SLO는 기준점으로 유지

- **ClusterOOMsTracker (failures = 0), SystemPodMetrics (restartCount = 0)** 는 4 run 모두에서 **안정적으로 0**이었고, 검증 문서에서 이미 **reasonable**하다고 평가했다.
- 이 두 지표는 **변동성이 거의 없고**, “클러스터 안정성·워크로드 건강도”의 **기준점**으로 그대로 두는 것을 권장한다. 레이턴시 SLO를 provisional로 두어도, OOM·재시작 SLO는 **안정적인 참조**로 유지하면 된다.

---

## 5. 요약

- **변동 원인:** 이미지 풀/cold start, kind 노드 리소스 압박, 스케줄링 지연은 **likely**. 스토리지/네트워크·컨트롤 플레인은 **possible**. first-run vs later-run·warm-up은 **unclear** 또는 **possible**.
- **추가 실험:** 이미지 preload + 연속 run, warm-up 후 측정, 시나리오 규모 축소, 노드/컨트롤 플레인 메트릭 수집이 원인 분리에 도움이 됨.
- **SLO 가설 수정 방향:** (1) local/kind와 실제 클러스터 수용을 **구분**해 환경별 SLO 정당화, (2) 로컬용 **임시 베이스라인 SLO**를 데이터 기반으로 두기, (3) 레이턴시 SLO는 **provisional**로 두고, 환경·실험이 개선되면 확정.
- **OOM·재시작 SLO:** 그대로 **기준점**으로 유지하며, 이미 reasonable함을 검증에서 확인한 상태임.

---

*레이턴시 변동성 분석. 최종 프로덕션 SLO 수치는 정의하지 않음.*
