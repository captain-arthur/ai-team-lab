# critical-review.md (날카로운 재검토: CAT + Monitoring)

이 문서는 요약이 아니라, 지금 구조가 **실무에서 계속 굴러갈 때 생길 문제**를 먼저 드러내기 위한 문서다.

---

## 1) 전체 구조 평가: 방향이 맞는가?
방향은 “파일 규약 + adapter 정규화 + 운영 대시보드”로 단순화하려는 시도인데, 실제로는 두 축이 **서로 다른 권위 체계**를 유지한다.
- CAT의 권위: `tool exit code` (CAT이 재판정하지 않음)
- Monitoring의 권위: Prometheus 데이터(상태 점수 0/1/2를 Q1/Q2/Q3로 변환)

문제는 “CAT과 Monitoring이 같은 의사결정 모델을 공유한다”는 전제가 문서에서만 성립하고, 연결(파이프라인/표준 스키마/매핑 규칙)이 구현 레벨로 고정되어 있지 않다. 즉, 운영자는 대시보드만 보고 CAT 결과는 ‘증거 저장’ 역할로 전락할 가능성이 크다.

결론: “구조적으로 맞다”는 말은 할 수 있지만, 실무 도입에서 가장 먼저 부딪히는 건 **권위의 분리로 인해 발생하는 운영 일관성 붕괴**다.

---

## 2) CAT에 대한 냉정한 평가

### 2.1 adapter 구조의 한계
Adapter를 얇게 유지하고 raw→표준만 하겠다는 계약은 좋은데, 현실에서는 다음이 반복된다.
- “얇음”을 유지하려면 tool별 raw에서 필요한 값을 찾는 비용이 adapter로 이동한다.
- 시간이 지나면 adapter가 사실상 “미세 SLI 정의/필터링/예외처리”까지 떠안는다.
- 지금 문서/코드에서도 ready/error latency처럼 운영 판단에 필요한 파생(예: 어떤 rcode를 실패로 볼지)이 대시보드 쪽에서 재구현될 가능성이 높다.

즉, adapter 계약이 “추가 재판정 금지”를 막아도, **SLI 정의의 책임 경계**가 시간이 지나며 흐려진다.

### 2.2 tool dependency 리스크
현재 CAT은 k6/Ginkgo를 중심으로 “정규화 가능한 흐름”이 증명됐고, CL2는 구조상 편입 가능하지만 파싱/실행의 완결성이 문서에서 핵심 계약 수준으로만 보인다(최소 구현 증명 부족).
- tool metric/raw 스키마가 바뀌면 adapter가 깨진다.
- 운영팀이 CL2/Ginkgo/k6 중 하나만 “항상 동작한다”고 믿게 되면, 나머지는 실제 운영에서 제외될 가능성이 높다(결국 tool lock-in).

### 2.3 SLO/SLI 통합 방식의 문제점
CAT 스펙은 `selected_sli` 키 네이밍 규칙을 강제하고, PASS/FAIL은 exit code로 고정한다. 하지만 Monitoring은 PromQL 기반으로 또 다른 SLO/임계값을 가진다.
- CAT의 SLO 정의(어댑터가 반영하는 임계값)와 Monitoring의 SLO 임계값이 동일한 근거로 연결되어 있지 않다.
- 결과적으로 동일한 컴포넌트라도 CAT은 PASS인데 Monitoring은 FAIL(또는 반대) 상황이 생길 수 있고, 운영자는 어느 쪽을 믿어야 하는지 다시 묻게 된다.

### 2.4 orchestration의 실제 필요성
Runner/Orchestrator가 최소라 하더라도, 파일 기반과 계약서가 없으면 팀은 결국 “각자 runbook/스크립트/파싱”을 만들어 분기한다.

하지만 더 근본적인 질문이 필요하다.
- 지금 CAT Orchestrator가 제공하는 가치는 “실행 자동화”인가 “표준 evidence 형식”인가?
- 만약 evidence 형식만 필요하다면, Orchestrator가 없어도 각 tool이 바로 `cat-result.json`을 만들게 해서 러너를 없애는 것이 더 단순할 수 있다.

👉 질문: 이걸 실제로 계속 운영할 수 있는가?  
대답은 “가능하지만”, 운영 관점에서 지속 비용은 낮지 않다. 계약서가 팀의 규율을 강제하는 형태라, 시간이 지나면 계약 준수 실패로 변형될 확률이 높다.

## 3) Monitoring에 대한 냉정한 평가

### 3.1 3가지 질문 구조의 장점/한계
장점:
- Row 3가 “상태→행동 번역기”로 고정되어, dashboard를 분석 플랫폼으로 키우지 않으려는 의도가 명확하다.

한계(실무 리스크):
- Q1/Q2/Q3의 ‘번역 규칙’이 PromQL에 직접 박혀 있다. 즉, 운영 기준이 바뀌면 9개 패널의 쿼리를 동시에 수정해야 한다.
- Row 3가 번역기라고 해도, 현재 구현은 action score를 만드는 과정에서 여러 조건을 재조합한다. “새 계산 금지”는 철학이지만, 실제 식은 이미 계산 중심이다.

### 3.2 골든 시그널 4개의 적절성
표면적으로 최소 4개인데, 실제로는 다음 위험이 있다.
- DNS에서 SERVFAIL만으로 “사용자 영향”을 충분히 대표하는가? NXDOMAIN/timeout/REFUSED 등은 배제된다.
- coredns restarts는 ‘원인’이라기보다 ‘증거’인데, 이를 Q2(내일 위험)로 승격시키는 논리적 근거가 데이터로 검증되지 않았다.
- istio의 request_rate는 golden signal로 포함됐지만, 현재 dashboard JSON에서는 request_rate가 Q1에서는 사실상 사용되지 않고(식에서 조건이 일부 누락/중복 가능), Q2/Action에서만 부분적으로 반영된다. 즉, “골든 시그널의 계약”과 “구현의 반영 방식”이 완전히 정렬돼 있지 않을 수 있다.

### 3.3 Q1/Q2/Q3 구조의 실무 적합성
실무에서 운영자는 5초 내 분류를 원하지만, 현재 PromQL은 사람이 읽기 어렵고(괄호/조건 중복), Grafana는 실패 시 “왜 실패인지”를 자동으로 설명하지 않는다.
- 번역기라고 해도 operator가 “어떤 조건 때문에 2가 됐는지”를 trace하기 어렵다.

### 3.4 실제 운영자가 사용할 가능성
가능성은 있다. 하지만 사용성이 생기는 조건이 붙는다.
- DNS plugin metrics가 실제로 동일한 라벨/스키마로 제공되어야 한다.
- istio destination_workload 라벨과 kube pod label/pattern이 실제 환경과 맞아야 한다.

이 조건이 깨지면, dashboard는 ‘문제해결 도구’가 아니라 ‘수정이 필요한 쿼리 모음’으로 전락한다.

👉 질문: “이 대시보드를 매일 볼까?”  
매일 보려면 “데이터 없음/쿼리 에러/라벨 mismatch”가 거의 없어야 한다. 현재 문서가 그 리스크를 충분히 줄였는지(특히 metric naming/label 확정)에 대한 검증이 없다.

## 4) 가장 큰 리스크 Top 5
1. **CAT ↔ Monitoring 권위 불일치**  
   CAT은 exit code 기반, Monitoring은 PromQL 기반. 동일한 의사결정을 보장하는 연결 고리가 없다.
2. **Metric/Label drift 취약성**  
   DNS plugin metrics 라벨(`k8s_app`, `rcode`, forward 계열)과 istio 라벨이 환경마다 다를 수 있다.
3. **PromQL 유지보수 비용 폭증**  
   Q1/Q2/Q3 및 Action score가 모두 쿼리 내부에 재조합되어, 기준 변경 시 9패널 동시 수정이 필요해진다.
4. **Golden signal의 운영 대표성 미검증**  
   SERVFAIL 중심, restarts 중심이 “사용자 체감 영향”과 일대일로 매핑되는지 데이터 검증이 문서화되지 않았다.
5. **의사결정 설명 가능성 부족**  
   state score만 보이고, 어떤 조건이 트리거했는지 operator가 바로 확인하기 어렵다.

## 5) 더 단순한 대안 제시 (중요)

### 5.1 CAT가 꼭 필요한가?
- 이미 운영 환경에서 “서비스 SLI/SLO”를 Prometheus/Evidence로 지속 측정하고, 롤링/배포 전에 충분한 canary/blackbox 체크가 있다면, CAT은 “필수”가 아니라 “추가 안전장치”다.
- 그 경우 CAT을 유지하더라도 Orchestrator/adapter를 플랫폼 수준으로 키울 필요가 없다.

더 단순한 선택지:
1. tool별로 `cat-result.json`을 직접 생성하게 만들어 runner를 제거(또는 최소 스크립트로 축소)
2. adapter는 raw→cat-result만 하고, SLO 판정 로직(임계값/필터링)은 도구 내부 또는 별도 단일 규칙으로 고정
3. evidence export는 runner 이후 한 번만(선택적) flatten해서 끝

### 5.2 Monitoring에서 더 단순하게 만드는 방법
현재는 “대시보드 쿼리에서 점수화”가 일어난다. 더 단순한 방식:
1. PromQL을 dashboard에서 빼고, `recording rules`로 `*_state_score`(또는 `*_dns_error_rate`, `*_dns_p95_ms` 등)만 만들어 둔다.
2. dashboard는 state score만 display(= Stat panel은 값만 보여주고 계산은 제거)
3. Row 3 번역 규칙도 recording rule로 만들어 “식 수정 지점”을 1곳으로 축소

이렇게 하면 “쿼리 복잡도/유지보수”가 가장 큰 리스크인 부분을 구조적으로 제거한다.

## 6) 한 단계 더 높은 추상화(본질)

### CAT의 본질은 무엇인가?
CAT의 본질은 “도구를 실행해 결과를 모으는 것”이 아니라, **표준 evidence 인터페이스를 통해 같은 종류의 결정을 반복 가능하게 만드는 것**이다.

따라서 본질에 맞는 최소 구현은 다음이어야 한다.
- 결정에 필요한 최소 evidence만 표준 구조로 저장
- 권위 모델(어떤 결과가 ‘최종 판단’인지)을 한 곳에서 고정
- 장기적으로 비교/누적이 가능한 key/tag을 반드시 포함

### Monitoring의 본질은 무엇인가?
Monitoring의 본질은 **현재/미래 위험을 사용자(운영자)의 언어로 번역하고, 다음 행동을 단일 선택으로 좁히는 것**이다.

그런데 현재 구현은 번역 결과는 보여주지만, operator가 다음 행동을 “근거 기반으로 선택”할 만큼의 설명가능성이 부족하다.

### 우리가 진짜 만들고 있는 것은 무엇인가?
- 사실상 “CAT 결과의 표준화”와 “DNS/입구 품질 상태 기계적 분류” 두 가지를 만들고 있다.
- 이 둘이 결합되어 ‘의사결정 파이프라인’을 만들었는지(= CAT→Monitoring 연결)가 없다면, 프로젝트는 분리된 산출물이 된다.

## 7) 최종 결론(필수 답변)
1. 이 프로젝트는 성공적인 방향인가?  
   - “방향 자체는 유효”하지만, **운영 의사결정 일관성(CAT vs Monitoring 권위 연결)**과 **유지보수성(복잡 PromQL)**에서 실패할 확률이 높다.
2. 지금 상태에서 바로 실무 도입 가능한가?  
   - 제한적으로는 가능하다. 단, metric/label 스키마가 환경에서 확정되고, recording rule로 계산 분리가 되지 않는다면 장기 운영은 비용이 커진다.
3. 가장 먼저 적용해야 할 것은 무엇인가?  
   - Dashboard의 PromQL 계산을 recording rules로 분리하고, CAT과 Monitoring이 동일한 SLI/SLO 정의(임계값/실패 정의)를 참조하도록 연결하는 작업.
4. 무엇을 버려야 하는가?  
   - 중복된 설계 문서/철학 문서가 아니라, **대시보드 쿼리 내부에 점수화 로직을 직접 박는 방식**(recording rule로 이동)이 버려져야 한다.
   - CAT에서 adapter의 책임 경계가 drift 되는 순간(예: 예외 처리/필터링이 adapter에 누적) 역시 버려야 할 방향이다(단일 규칙으로 이동 필요).

## 버릴 것(명시)
- “CAT 결과와 Monitoring 상태가 동일한 권위를 가진다”는 암묵 가정  
  - 이유: 실제 연결(스키마/매핑/파이프라인) 없이는 운영에서 신뢰가 깨진다.
- PromQL 내부 재조합으로 만드는 Action score  
  - 이유: 기준 변경 시 9패널을 동시 수정해야 하는 유지보수 지옥이 된다.
- 대시보드 철학 문서 중 최종 계약을 대체하는 문서  
  - 이유: 최신화된 final lock/PromQL spec/JSON이 이미 권위다.

## Final Question(요구사항 답변)
👉 “이 프로젝트에서 가장 쓸데없는 것은 무엇이며, 왜인가?”
- 가장 쓸데없는 것은 **“CAT이든 Monitoring이든 권위 모델을 문서로만 합의하고, 실제 연결 규칙(증거→상태→의사결정)이 구현으로 고정되지 않은 상태”**다.  
  이유: 운영자가 실제로 보는 것은 dashboard 상태 score와 runbook 연결인데, CAT 결과는 그 의사결정에 어떤 입력으로 들어오는지 불명확하면 결국 ‘있는 듯 없는 듯’ 방치된다(증거 저장은 늘지만 행동으로 이어지지 않는다).
