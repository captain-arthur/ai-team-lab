# 운영 모델 V2: Confidence / Risk / Action

이 문서는 “지금 안전한가 / 내일도 안전한가 / 무엇을 해야 하는가”를 대시보드의 row/panel로 옮기기 위한 운영 모델을 정의한다.

## 1) 운영 모델 정의(지금/내일/행동)

### A. 지금 안전한가(Confidence)
정의: 운영자는 “현재 사용자 영향이 허용 가능한 수준인지”를 단시간에 확인해야 한다.
- 판정 단위: 컴포넌트 단위(3개)
- 증거: 응답 품질(성공/실패/지연) + 컴포넌트 준비성/헬스
- 출력: `safe / watch / fail` 또는 `PASS / INVESTIGATE`처럼 즉시 행동 가능한 상태

### B. 내일도 안전한가(Risk)
정의: 현재는 괜찮아 보여도, “조기 악화가 이미 시작됐는지”를 확인한다.
- 판정 단위: 컴포넌트 단위(3개)
- 증거: 최근 추세(지연/오류율/타임아웃 증가), 그리고 자원/대기열 포화 징후
- 출력: `stable / risk-increasing / risk-high`처럼 24h가 아니라 “다음 운영 창(예: 내일 오전~오후)”에 대한 조사 필요도를 제공

### C. 지금 무엇을 해야 하는가(Action)
정의: 운영자가 다음 5~10분 안에 할 수 있는 “다음 행동”을 명확히 제시한다.
- 판정 단위: Row의 상태에 따라
- 증거: Confidence/Risk 상태 + 컴포넌트 특성에 맞춘 1차 원인 후보(예: upstream 문제 vs 내부 포화)
- 출력: “다음으로 무엇을 볼 것인가”를 runbook 수준으로 연결

요약: 이 모델은 metric을 모아 보여주기 위한 모델이 아니라, **판정 가능한 최소 신호 묶음이 만들어내는 운영 상태**를 목표로 한다.

## 2) 이를 대시보드 구조(Row / Panel)로 옮기는 방법

### Row 구조(권장 3행, 패널 수 최소)
1. Row 1: `지금 안전한가(Confidence)`
   - 역할: 현재 상태를 운영자가 즉시 판정하도록 만든다.
   - 패널: 컴포넌트 3개 각각 1개(총 3개)
2. Row 2: `내일도 안전한가(Risk)`
   - 역할: 오늘의 괜찮음이 내일에 깨질 조짐을 잡는다.
   - 패널: 컴포넌트 3개 각각 1개(총 3개)
3. Row 3: `지금 무엇을 해야 하는가(Action)`
   - 역할: 상태에 따라 조사 우선순위를 정하고(runbook 진입), 운영자가 5~10분 내 행동을 시작하도록 한다.
   - 패널: 컴포넌트 3개 각각 1개(총 3개)

### 각 Row의 역할(중요 원칙)
- Row 1은 “해석이 아니라 판정”이 목표다.
- Row 2는 “지금이 아니라 추세”가 목표다.
- Row 3는 “다음 행동(조사/완화)”이 목표다.

즉, 동일한 metric을 반복하지 않고도 화면 구조가 의미를 유지한다.

## 3) 기존 ‘압박 → 부담 → 장애’ 체인의 평가

이 체인은 설명에는 강하지만, 운영자가 “지금 무엇을 해야 하는가”에 빠르게 도달하기에는 두 가지 리스크가 있다.

1) 체인이 늘 “측정 가능”하지 않다
   - 압박(usage)은 보이는데, 부담(서비스 품질로 전이되는 지점)이 지연돼 관측된다.
2) 운영 행동이 흐려진다
   - 장애까지 이어지는 chain이 길어질수록, 운영자가 “어느 단계에서 어떤 조치를 해야 하는지”가 뚜렷해지기 어렵다.

다만, 이 모델이 완전히 쓸모없다는 뜻은 아니다.
- 대시보드 내부에서 chain을 “메인 판정 원리”로 쓰는 것은 위험하다.
- 대신, Risk Row에서 “전이(usage→quality)”를 설명하는 보조 개념으로 제한적으로 활용하면 유효하다.

## 4) 대안 모델 제안(후보)

후보 1) Safety / Risk / Action 모델
- 장점: 질문(지금/내일/무엇)과 자연스럽게 1:1 대응된다.
- 단점: Risk의 정의가 애매해지면 다시 metric 나열로 돌아갈 수 있다.

후보 2) Decision / Evidence / Action 모델
- 장점: “무엇이 주도하는가”를 decision으로 고정한다.
- 단점: Evidence 수준이 너무 구체화되면 과설계가 된다.

후보 3) Confidence / Warning / Investigation 모델
- 장점: 운영자의 심리(확신/조사 필요)를 정확히 겨냥한다.
- 단점: Warning을 ‘경보’로 오해하면 alert 정책 설계로 흐를 수 있다.

후보 4) TDD를 직접 번역한 구조(테스트→증거→판정)
- 장점: 판정 단위가 명확해진다.
- 단점: 테스트가 많아지면 운영 대시보드가 실험 플랫폼처럼 변질될 수 있다.

## 5) 최종 모델 선택

채택: **Confidence / Risk / Action** (Safety / Risk / Action을 confidence 언어로 정교화)

채택 이유:
- 질문 3개가 row 구조와 정확히 맞물린다.
- 각 row가 목표가 다르기 때문에, 동일 metric 반복 없이도 의미가 유지된다.
- Risk를 “추세 기반 조사 필요도”로 제한하면 과설계 위험이 낮다.

버리는 것(중요):
- ‘압박→부담→장애’를 메인 체인으로 계속 그리는 방식
  - 대신, Risk Row에서 “전이 징후(quality 저하로 가는 추세)”만 잡고, chain을 자세히 설명하지 않는다.
