# 프로젝트 브리프: 핵심 컴포넌트 운영 확신 대시보드 설계

## 이번 프로젝트의 목적
Kubernetes 운영에서 metric/패널이 많아도 운영자는 여전히 다음 질문에 답하지 못한다.
- “지금 안전한가?”
- “내일도 안전한가?”
- “지금 무엇을 해야 하는가?”

이 프로젝트는 `istio-ingressgateway`, `nodelocaldns`, `coredns` 세 컴포넌트에 대해, **운영 판단과 행동(runbook 진입)**이 한 화면에서 이어지도록 Grafana 대시보드 설계 원칙과 패널 구체안을 확정한다.

## 왜 이 3개 대상이 중요한가
- `istio-ingressgateway`: 서비스로 들어오는 트래픽의 관문. 여기 이상은 외부 트래픽 실패/지연으로 바로 연결될 수 있다.
- `nodelocaldns`: 노드 로컬 캐시 DNS. 여기 이상은 “이름 해석 실패/지연”으로 다수 서비스의 연쇄 장애를 유발한다(빠르게 광범위해질 수 있음).
- `coredns`: 클러스터 DNS의 근간. 재귀/헬스/리소스 이슈는 내부 서비스 통신의 기초를 흔든다.

즉, 이 세 컴포넌트는 운영자가 “안전/위험”을 판단할 때 **확신을 얻어야 하는 핵심 의존성**이다.

## 무엇을 최종적으로 산출할 것인지
1. 운영 관점 이해 문서(각 컴포넌트의 운영적 증상과 우선 걱정 포인트)
2. “무엇이 대시보드를 주도해야 하는가?”에 대한 설계 원칙(운영 모델 + TDD/Kent Beck 철학의 실제 번역)
3. “지금 안전/내일 안전/무엇을 할 것”을 row/panel로 옮긴 구체 대시보드 청사진
4. Evidence(관측) 관점에서 설계의 실무 타당성 리뷰

## 성공 기준
다음이 모두 충족되어야 성공이다.
1. 문서만 보고도 **실제로 Grafana row/panel을 그릴 수 있을 정도로 구체적**이어야 한다.
2. 각 패널이 “판단 → 행동”으로 이어져야 한다(이상 징후가 보이면 무엇을 먼저 확인할지).
3. 기술적 타당성과 운영적 타당성이 함께 검증되어야 한다.
4. “압박 → 부담 → 장애” 같은 장애 체인 아이디어는 무조건 유지하지 않고, **실무 유효성이 낮으면 과감히 대체**한다.
