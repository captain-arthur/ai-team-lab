# 대시보드 블루프린트(바로 그릴 수 있는 수준)

이 설계는 Row 단위로 운영 질문(지금/내일/무엇을)을 고정하고, 각 Row의 패널이 그 질문에 대해 “판정 → 다음 행동”을 만들도록 한다.

## 1) 최종 추천 대시보드 구조

총 3 Row, 총 9 Panels(컴포넌트당 Row 3개, 각 Row는 패널 1개씩)

1. Row 1: `지금 안전한가(Confidence)`  
   - 목적: 현재 사용자 영향이 허용 가능한지 즉시 판정  
   - 패널 수: 3개(istio-ingressgateway / nodelocaldns / coredns 각각 1)
2. Row 2: `내일도 안전한가(Risk)`  
   - 목적: 현재 이상이 내일로 전이될 추세가 있는지 확인  
   - 패널 수: 3개(동일)
3. Row 3: `지금 무엇을 해야 하는가(Action)`  
   - 목적: 현재 상태에서 “다음 5~10분 조치”를 시작  
   - 패널 수: 3개(동일)

## 2) Row 1 패널 설계: 지금 안전한가(Confidence)

### Panel 1: `istio-ingressgateway 안전도(현재)`
- 왜 중요한가: 외부 사용자 관측 품질(성공률/지연)이 가장 먼저 깨지는 지점이기 때문이다.
- 어떤 질문에 답하는가: “지금 안전한가?”
- 패널 형태: `Stat + Sparkline(최근 5~10m)`
- 기본 판정 기준(초안):
  - `safe`: 최근 5분 동안 요청 타임아웃 비율이 낮고(Timeout/5xx 계열이 안정), p95 지연이 운영 기준치 이내이며, 게이트웨이 준비성(Ready/Health)이 정상
  - `watch`: 위 조건 중 1개가 약하게 초과(지연/오류가 소폭 상승)하거나 Ready 신호가 간헐적으로 흔들림
  - `fail`: p95 지연 또는 타임아웃/에러율이 지속적으로 운영 기준치를 넘고(최소 2개 연속 윈도우), Ready/Health가 불안정
- 이상 시 운영자가 무엇을 해야 하는가:
  - watch: 게이트웨이 로그에서 upstream 연결 실패/라우팅 오류 징후를 먼저 확인하고, 영향 경로(도메인/게이트웨이 서버/경로)를 좁힌다.
  - fail: “게이트웨이 자체 문제 vs upstream 문제”를 분리한다. 즉시 (1) 게이트웨이 헬스/리소스, (2) upstream 엔드포인트/서비스 상태를 순서대로 확인하고 대응한다.

### Panel 2: `nodelocaldns 안정성(현재)`
- 왜 중요한가: 로컬 DNS 지연/실패는 재시도 폭발로 다수 서비스에 연쇄 영향을 준다.
- 어떤 질문에 답하는가: “지금 안전한가?”
- 패널 형태: `Stat + 분포/히스토리(최근 5~10m에서 timeout 지표 집중도)`
- 기본 판정 기준(초안):
  - `safe`: 노드 로컬 DNS 질의가 대부분 성공하며, 타임아웃 비율과 응답 p95가 기준 이내, liveness/ready가 정상
  - `watch`: 응답 p95 또는 timeout이 소폭 상승하거나 특정 노드군에서만 편차가 나타남
  - `fail`: timeout/실패가 특정 노드군을 넘어 “여러 노드”로 확산되고, 서비스에서 이름 해석 실패가 증가하는 패턴이 동반
- 이상 시 운영자가 무엇을 해야 하는가:
  - watch: upstream(주로 coredns) reachability를 먼저 확인하고, 네트워크 정책/노드 라우팅 이슈 가능성을 점검한다.
  - fail: coredns 의존성이 깨졌는지부터 확인하고(상위 DNS 헬스), nodelocaldns 포드 상태(재시작/리소스 압박)를 확인한 뒤, 필요한 경우 노드군 단위 롤백/재시작을 검토한다.

### Panel 3: `coredns 안정성(현재)`
- 왜 중요한가: DNS 근간(재귀/헬스/플러그인 체인). 여기 문제가 생기면 전체가 흔들릴 가능성이 높다.
- 어떤 질문에 답하는가: “지금 안전한가?”
- 패널 형태: `Stat(health) + 타임시리즈(에러/지연의 짧은 창)`
- 기본 판정 기준(초안):
  - `safe`: coredns 헬스 엔드포인트가 정상이고, DNS 응답 에러(타임아웃/SERVFAIL 계열)가 낮으며, 응답 지연이 안정적
  - `watch`: 에러 유형 중 특정 플러그인/업스트림 경로에서 실패가 늘거나, 지연이 점진적으로 증가
  - `fail`: 헬스 불안정, 에러율 급상승, 또는 재시작/구성 로드 실패 같은 운영 관측 이벤트가 동반
- 이상 시 운영자가 무엇을 해야 하는가:
  - watch: coredns forward/upstream 응답 품질을 확인한다(상위 DNS 또는 네트워크 경로).
  - fail: 서비스 영향이 빠르게 전파되므로, (1) 헬스/로그에서 근본 원인(업스트림, 설정 로드, 리소스)을 우선 분류하고, (2) 즉시 완화(스케일/재시작/설정 롤백)를 runbook 순서로 진행한다.

## 3) Row 2 패널 설계: 내일도 안전한가(Risk)

Row 2의 핵심은 “추세”다. 지금 safe라도 추세가 악화면 risk로 분류한다.

### Panel 4: `istio-ingressgateway 위험도(내일)`
- 왜 중요한가: 게이트웨이 문제는 사용자 관측 실패로 빠르게 전이되므로, 추세 조기 경보가 가치가 크다.
- 어떤 질문에 답하는가: “내일도 안전한가?”
- 패널 형태: `Time series(최근 1~6h) + Risk state Stat`
- 위험 판정 기준(초안):
  - `stable`: p95 지연과 타임아웃/에러율이 최근 창에서 거의 평탄
  - `risk-increasing`: 지연 또는 타임아웃이 “완만하지만 지속적으로” 증가(예: 2개 이상의 연속 윈도우에서 회복 실패)
  - `risk-high`: 안전 기준에 근접하거나 반복적으로 경계선을 넘고, 게이트웨이 준비성/회로차단 지표가 동반 악화
- 이상 시 운영자가 무엇을 해야 하는가:
  - risk-increasing: upstream/라우팅/인입 트래픽 패턴 변화 여부를 확인하고, 용량(HPA/리소스)과 정책(라우팅/타임아웃) 변경 필요성을 검토한다.
  - risk-high: 사전 완화(스케일, 라우팅 우회, 타임아웃/서킷 브레이커 조정)를 runbook으로 준비한다.

### Panel 5: `nodelocaldns 위험도(내일)`
- 왜 중요한가: 로컬 DNS는 “일시적 이상”이 아니라 “점진적 누적”이 문제를 키우는 경우가 많다(캐시/포워딩/업스트림 변동).
- 어떤 질문에 답하는가: “내일도 안전한가?”
- 패널 형태: `Time series(노드 편차 포함) + Risk state Stat`
- 위험 판정 기준(초안):
  - `stable`: timeout/지연이 낮고 분산이 안정적(노드 편차 없음)
  - `risk-increasing`: 특정 노드군에서 지연/timeout이 늘고, 시간이 지나며 영향 노드가 확대
  - `risk-high`: coredns upstream 관련 실패 징후가 동반되며, 전체 쿼리 성공률이 하락 추세
- 이상 시 운영자가 무엇을 해야 하는가:
  - risk-increasing: coredns 헬스와 upstream 경로를 우선 확인하고, 노드군 단위 원인(네트워크 정책/노드 리소스)을 좁힌다.
  - risk-high: 내일 운영 시간대에 이름 해석 품질 저하가 예상되므로, 사전 조치(스케일/재시작/구성 롤백)를 준비한다.

### Panel 6: `coredns 위험도(내일)`
- 왜 중요한가: coredns는 리소스/큐잉/업스트림 의존성이 흔들리면 지연과 에러가 누적된다.
- 어떤 질문에 답하는가: “내일도 안전한가?”
- 패널 형태: `Time series + Risk state Stat`
- 위험 판정 기준(초안):
  - `stable`: 응답 지연과 에러율이 낮고, 헬스/준비성 이벤트가 없음
  - `risk-increasing`: 에러율 또는 지연이 “회복 없이” 상승하며, 재시도/큐잉 징후가 증가
  - `risk-high`: 재시작/헬스 불안정이 반복되거나, 리소스 압박 신호가 지속
- 이상 시 운영자가 무엇을 해야 하는가:
  - risk-increasing: upstream 문제(외부 forward, 네트워크) 또는 설정 변경 영향 여부를 확인한다.
  - risk-high: 확실한 완화(스케일/롤백/구성 점검)를 내일 전에 준비한다.

## 4) Row 3 패널 설계: 지금 무엇을 해야 하는가(Action)

Row 3는 “조사 순서”를 고정한다. 즉, 동일한 질문(무엇을 해야 하는가)에 대해 컴포넌트별로 가장 먼저 확인할 것을 3~5단계로 제공한다.

### Panel 7: `istio-ingressgateway 조사/조치(현재 상태 기준)`
- 어떤 질문에 답하는가: “지금 무엇을 해야 하는가?”
- 패널 형태: `Table(상태 → 1차 조사 → 2차 조사 → 실행/완화)`
- threshold/판정 기준(초안): Row 1의 `safe/watch/fail`과 Row 2의 `stable/risk-increasing/risk-high` 조합으로 상태를 매핑한다.
  - `fail`이면 “즉시 조사/완화(첫 5~10분 내)”에 해당
  - `watch`면 “원인 분류(로그/업스트림)부터 시작”에 해당
  - `safe`면 “내일 준비(링크/모니터링 유지)”에 해당
- 상태 기반 행동(초안):
  - Confidence fail: (1) 게이트웨이 헬스/리소스, (2) upstream 엔드포인트 상태, (3) 라우팅/인증(특히 TLS) 오류 징후, (4) 필요시 스케일/우회
  - Confidence watch: (1) 로그에서 실패 유형 분류, (2) 특정 도메인/경로 편중 여부, (3) upstream 병목 가능성 확인
  - Confidence safe: runbook “예방 확인(내일 Risk Row에서 링크)”만 제공

### Panel 8: `nodelocaldns 조사/조치(현재 상태 기준)`
- 어떤 질문에 답하는가: “지금 무엇을 해야 하는가?”
- 패널 형태: `Table`
- threshold/판정 기준(초안): Row 1의 `safe/watch/fail`이 1차 트리거, Row 2의 risk 상태는 “내일 준비 정도”를 보강한다.
- 상태 기반 행동(초안):
  - Confidence fail: (1) coredns upstream reachability 확인, (2) nodelocaldns 포드 재시작/리소스 압박 확인, (3) 노드군 네트워크/정책 점검, (4) 필요시 노드군 단위 완화
  - Confidence watch: (1) 노드 편차 확인, (2) upstream 실패 유형 분류, (3) 캐시/포워딩 경로 점검
  - Confidence safe: 내일 Risk Row 기반 “경고만” 유지

### Panel 9: `coredns 조사/조치(현재 상태 기준)`
- 어떤 질문에 답하는가: “지금 무엇을 해야 하는가?”
- 패널 형태: `Table`
- threshold/판정 기준(초안): Row 1의 `safe/watch/fail`에서 “첫 행동”을 고정하고, Row 2의 `risk-high`이면 “확실한 완화 준비”를 우선한다.
- 상태 기반 행동(초안):
  - Confidence fail: (1) coredns 헬스/로그에서 근본 원인 분류(업스트림/설정 로드/리소스), (2) upstream 의존성(외부 forward 또는 내부 라우팅) 확인, (3) 즉시 완화(스케일/재시작/롤백)
  - Confidence watch: (1) 에러 유형의 변화 추세, (2) forward upstream 품질 확인, (3) 구성 변경 히스토리 교차검증
  - Confidence safe: 내일 Risk Row에서 경고 트리거 가능성을 모니터링
