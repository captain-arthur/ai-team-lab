# 최종 보고서: 운영 확신 기반 핵심 컴포넌트 대시보드

## 1) 최종 추천 운영 대시보드 방법론
운영 대시보드는 “관측”이 아니라 “판정과 행동”을 만들도록 설계해야 한다.
이번 프로젝트의 방법론은 **Confidence / Risk / Action** 3단계로 고정한다.

- Row 1(Confidence): 지금 안전한가?를 컴포넌트 3개 단위로 즉시 분류한다.
- Row 2(Risk): 내일도 안전한가?를 추세 기반으로 승격한다(안정/악화 조짐/위험).
- Row 3(Action): 현재 분류 상태에 따라 5~10분 내 조사/완화가 시작되도록 “조사 순서”를 제공한다.

## 2) “무엇이 대시보드를 주도해야 하는가?” 최종 답
**confidence-driven + decision-driven(판정 단위)**

TDD가 “테스트(작은 단위)가 주도한다면”, 운영 대시보드는 “증거 기반 판정(작은 단위 confidence)이 주도”해야 한다.
metric은 도구이며, 주도 원리는 “운영자가 결정할 수 있는 형태의 확신”이어야 한다.

## 3) 왜 이 방법론이 적절한가
- 운영자의 질문(지금/내일/무엇을)이 row 구조와 1:1 대응된다.
- ‘압박 → 부담 → 장애’처럼 긴 chain 설명은 메인 판정으로 쓰지 않고, Risk에서 필요한 전이 징후만 제한적으로 사용한다.
- metric 나열이 아니라, “상태 분류 + 조사 순서”로 행동까지 연결된다.

## 4) 대상 3개 컴포넌트에서 무엇을 봐야 하는가(요약)
- `istio-ingressgateway`
  - 지금: 사용자 관측 성공/타임아웃/지연과 게이트웨이 준비성
  - 내일: 오류/지연의 추세와 서킷차단/완화 신호의 반복
  - 행동: upstream vs 게이트웨이 헬스 분기 후, 라우팅/연결 실패 유형부터 분류
- `nodelocaldns`
  - 지금: 노드 로컬 DNS의 질의 성공/타임아웃과 응답 지연
  - 내일: upstream(coredns) reachability 악화와 캐시 효과 저하 추세
  - 행동: coredns upstream 우선 확인 → nodelocaldns 안정성/리소스 → 노드군 네트워크/정책 순서
- `coredns`
  - 지금: 헬스/준비성과 DNS 응답 에러/지연 안정성
  - 내일: 큐잉/워커 소진, 재시작/헬스 불안정 반복, forward/upstream 품질 악화 추세
  - 행동: 원인 분류(업스트림/설정/리소스) 후 스케일/재시작/롤백을 runbook 순서로 진행

## 5) 최종 대시보드 구조 요약
총 3 Row / 9 Panels
- Row 1: istio-ingressgateway / nodelocaldns / coredns (각 1개)
- Row 2: istio-ingressgateway / nodelocaldns / coredns (각 1개)
- Row 3: istio-ingressgateway / nodelocaldns / coredns (각 1개, 상태→조사 순서 테이블)

## 6) 바로 구현 가능한 수준인가?
네, “바로 구현 가능한 수준”이라고 판단한다.
- Row/panel 목적과 판정 기준(초안)이 문서에 들어가 있고,
- 패널 유형(Stat/Time series/Table)이 명시돼 있으며,
- 이상 시 운영자가 무엇을 먼저 할지까지 고정돼 있다.

단, Risk Row의 추세 판정 기준(윈도우/승격 조건)은 실제 운영 환경에 맞춰 최소 1회 튜닝이 필요하다.

## 7) 최종 결론
이 결과물을 기반으로 실제 운영 대시보드를 설계/구현할 수 있다.
왜냐하면 이 문서가 단순한 “철학”이 아니라, Row/panel/판정/행동을 직접 연결하는 수준으로 내려와 있기 때문이다.
