# Review Notes: k6를 써보고 판단

**Date:** 2026-03-18

## 1) 강점
- **Scenario 표현**이 단순하고 즉시 실행 가능(단일 요청/다단계 흐름/부하 패턴).
- **SLI가 즉시 나온다**: p95 latency, 실패율, 요청률을 실행 요약에서 바로 확인.
- **SLO 게이팅이 내장**: threshold 위반 시 **명확한 FAIL + 종료 코드**(예: p95<50ms 위반 → exit_code=99).

## 2) 한계
- 내부(클러스터) 관측은 기본 범위 밖이라, “외부 SLI 악화”의 원인 분류에는 **내부 신호(예: Prometheus)**가 필요해질 수 있음.
- 합격 기준이 내부 SLI(제어면/노드/리소스/재시작) 중심이면 k6는 “부하 발생”까지만 담당하고 **판정 주체가 되기 어렵다**.

## 3) 실제 써보면서 느낀 점(실행 기반)
- 출력이 **판정 중심**(threshold 섹션)이라 자동화에 유리.
- VU 기반/arrival-rate 기반은 “부하 정의 축”이 달라, CAT에서 **테스트 정의(목표가 VU인지 RPS인지)**를 명시하지 않으면 해석이 흔들린다.
- `sleep`(think time)은 latency를 늘리는 장치가 아니라, VU가 수행하는 iteration을 줄여 **달성 처리량**을 바꾸는 장치로 읽는 게 정확하다.

## 4) 적합한 상황
- CAT에서 “클라이언트 관측 SLI”로 합격 판정하는 테스트(예: ingress/서비스 앞단 HTTP 성능/오류율).

## 5) 부적합한 상황
- 합격 기준이 내부 SLI(컨트롤플레인/노드/리소스/재시작 등) 중심이거나, 브라우저 수준의 현실성이 필요한 경우.
