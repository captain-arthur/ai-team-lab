# Ginkgo 조사: 범위/평가축/제외 항목

**Date:** 2026-03-18

## 1) 조사 범위 정의
### 반드시 이해해야 하는 것(개념/구조)
- Ginkgo가 무엇인지(테스트 프레임워크로서의 역할)
- Ginkgo의 철학(왜 이런 구조를 선택했는가: BDD/명세 중심)
- 구성요소와 실행 모델(예: spec/group/runner가 언제 무엇을 실행하는지)
- Go 기반 커스텀 시나리오 구현 가능성(“테스트 코드”로 scenario/measurement/assertion을 묶을 수 있는지)
- 결과 포맷/리포트/PASS-FAIL 구조(프레임워크가 어떤 출력을 자동 생성하는지)

### 실제로 실험해야 하는 것(가능/판단을 위한 최소)
- “CAT 최소 조건 4개”를 Ginkgo로 **코드 수준에서 충족**할 수 있는지:
  1. Scenario Injection: 테스트 입력/설정으로 scenario를 주입(또는 외부 런타임에서 주입)할 수 있는가
  2. SLI 측정: 실행 중/후에 SLI 값을 산출/수집할 수 있는가
  3. SLO 평가: 명확한 단언으로 PASS/FAIL을 만들 수 있는가
  4. Result Persistence: PASS/FAIL과 핵심 측정값을 파일로 저장할 수 있는가
- 결과 저장이 “리포트만 남는 것”이 아니라 CAT에서 후속 소비가 가능한 파일 스키마로 남는가(최소 JSON/JUnit 등)

### 이번 단계에서 제외할 것(구현/통합 과대범위 방지)
- CAT 오케스트레이터 전체 구현(스케줄러/런 UI/대시보드)
- 실제 운영 클러스터에서의 장기 성능 측정(규모/튜닝)
- k6/CL2의 세부 내부 분석(대신 역할 경계는 간단히 확인)

## 2) 평가 축(Seven-axis, CAT 관점)
아래 각 축마다 “가능/불가능/부분 가능”을 판정한다.

1. **Scenario Injection 가능성**
   - 테스트 실행 시 외부 입력(config/env/flags)으로 scenario를 바꿀 수 있는가
   - 시나리오가 “Job 단위”로 묶이는가(테스트 케이스=Job으로 매핑 가능)
2. **SLI Measurement 가능성**
   - 측정 로직(HTTP/상태 확인/지표 수집)을 테스트 코드로 수행/집계할 수 있는가
   - 측정 결과를 구조화(키-값)해 다음 단계로 넘길 수 있는가
3. **SLO Evaluation / 명확한 PASS-FAIL 가능성**
   - assertion이 “SLO 게이트”로 동작할 수 있는가(애매한 경고/로그가 아니라 단언)
   - fail 이유(어떤 SLO/얼마나 위반)를 식별 가능한가
4. **Result Persistence 가능성**
   - 프레임워크 기본 리포트 외에, CAT가 요구하는 최소 파일 스키마(JSON/JUnit)를 남길 수 있는가
5. **완전 커스텀 시나리오 구현성**
   - 시나리오 생성/측정/단언을 Ginkgo spec 내부에서 완전히 커스텀할 수 있는가
   - “테스트가 너무 프레임워크 제약에 묶이지 않는가”
6. **다른 도구와 통합 가능성**
   - k6/CL2 결과를 입력으로 읽어, Ginkgo가 최종 PASS/FAIL을 단언하거나(또는 합성) 기록할 수 있는가
7. **오케스트레이션 기반으로 확장 가능성**
   - Job runner가 Ginkgo 실행을 조립해 “CAT run”의 일부로 만들 수 있는가
   - 결과 디렉터리 규약(결과 파일 저장) 유지가 가능한가
