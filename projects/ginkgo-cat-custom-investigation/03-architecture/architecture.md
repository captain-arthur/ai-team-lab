# Ginkgo 기반 Custom CAT 조사: 단계별 추진 계획 + 산출 구조

**Date:** 2026-03-18

## 1) 단계별 추진 계획(3~5단계, 구현 없이 조사/실험 후보 확정)
### Step 1. 개념/구조 고정(리서치)
- Ginkgo의 구성요소/실행 모델을 CAT 관점으로 매핑한다.
- 목표: “테스트(spec)가 곧 Job 단위가 될 수 있는지”를 먼저 확인한다.

### Step 2. CAT 최소 조건 매핑(가능성 판정용 체크리스트)
- 평가 축 1~7을 기준으로 “각 조건이 Ginkgo에서 어디에 걸리는지”를 문서화한다.
- 목표: 시나리오 주입/SLI 측정/SLO 단언/파일 저장이 Ginkgo에서 **기술적으로 가능한지**와 **얼마나 번거로운지**를 정리한다.

### Step 3. 최소 시나리오 프로토타입 설계(실험 후보 확정)
- 구현은 다음 단계에서 하되, 이번 단계는 “실험에 쓸 테스트 스펙(입력/측정/단언/파일)”을 2~3개 확정한다.
- 목표: “우리가 실제로 알고 싶은 것”을 직접 찌르는 실험을 만든다.

### Step 4. 결과 포맷 전략 확정(판정/저장)
- Ginkgo의 기본 리포트(JUnit 등)와 커스텀 출력(JSON 등) 중 무엇을 CAT에 쓸지 결정한다.
- 목표: CAT가 후속으로 소비 가능한 최소 파일 스키마가 나오는지 확인 계획을 세운다.

### Step 5. CAT 적합성 및 플랫폼성 평가(최종 판단 기준 고정)
- 다음 질문에 최종 YES/NO로 답한다.
  - Ginkgo로 “완전 커스텀 CAT Job”이 가능한가?
  - 결과 파일과 PASS/FAIL 단언이 CAT 최소 요구를 충족하는가?
  - k6/CL2와 함께 “도구 조합”으로 확장 가능한가?
  - 오케스트레이터 플랫폼 기반(런너/표준화)까지 이어질 수 있는가?

## 2) 최소 실험 후보(2~3개, 아직 구현하지 않음)
아래 후보는 “CAT 최소조건을 직접 찌르는지”가 핵심이다.

### 후보 A: 순수 Go 내부 측정 + SLO 단언(파일 저장 포함)
- 검증하려는 것
  - SLI를 테스트 코드에서 산출할 때, Ginkgo assertion이 SLO 게이트(PASS/FAIL)로 명확히 동작하는지
  - 결과를 JSON 파일로 저장해 CAT가 소비할 수 있는지
- 왜 Ginkgo로 해볼 가치가 있나
  - “프레임워크가 단언과 결과를 어떻게 패키징하는지”를 가장 직접적으로 본다
- k6/CL2와 어떤 차이가 드러나나
  - k6/CL2가 기본적으로 측정/판정을 “도구 모델”로 제공한다면, Ginkgo는 “테스트 코드가 전부”가 된다(완전 커스텀의 비용/제약 확인)

### 후보 B: Kubernetes 오브젝트 상태 기반 scenario(Conformance 성격)
- 검증하려는 것
  - Scenario Injection(외부 입력으로 scenario 조건을 바꾸기)이 가능한지
  - SLI를 클러스터 상태에서 읽고(예: 리소스 생성/ready 여부, 이벤트 유무 등) SLO로 단언할 수 있는지
  - 결과 persistence가 클러스터/상태 기반 테스트에서도 일관되는지
- 왜 Ginkgo 가치가 있나
  - Go 기반이므로 “클러스터 관측 로직 + 단언 + 파일 출력”을 통합할 수 있는지 확인
- k6/CL2 차이
  - k6는 외부 관측(HTTP 성능) 중심, CL2는 부하 시나리오/내부 관측 중심. Ginkgo는 “관측과 단언이 완전 커스텀”되는 축 확인

### 후보 C: k6/CL2 결과를 입력으로 받아 합성 PASS/FAIL 단언(통합성 실험)
- 검증하려는 것
  - 다른 도구 raw 결과를 파일로 읽고, Ginkgo가 최종 PASS/FAIL을 단언/기록할 수 있는지
  - CAT 통합을 “테스트 프레임워크” 단에서 구현할 수 있는지
- 왜 Ginkgo 가치가 있나
  - 오케스트레이션/플랫폼성 여부를 가르는 가장 중요한 통합 실험
- k6/CL2 차이
  - k6/CL2는 실행 도구, Ginkgo는 “합성/표준화된 판정 레이어” 역할을 맡을 수 있는지 확인

## 3) 최종 산출물 구조 제안(이 조사 종료 시 남길 것)
### 권장 프로젝트 구조
- `01-manager/` : 목적/질문/성공 기준
- `02-research/` : 범위/평가축/제외 항목(이번 단계 고정)
- `03-architecture/` : 추진 단계 + 최소 실험 후보 2~3개(스펙 수준)
- `04-engineering/` : 각 후보 실험의 “입력/측정/단언/파일 스키마”를 테스트 스펙으로만 정리(구현 금지)
- `05-review/` : 구현 난이도/리스크/대안 비교(실험 계획의 타당성 검토)
- `06-documentation/` : “무엇을 증명하려는 조사인지”와 최종 판단 프레임(다음 단계로 이어질 것)
