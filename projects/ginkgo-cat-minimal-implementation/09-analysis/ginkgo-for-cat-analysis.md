# Ginkgo 실전 CAT 분석(장단점/비교)

## 1) 실전에서 Ginkgo가 주는 장점
- 테스트가 “상태/조건/행동”으로 계층화되어 실패 지점 추적이 빠르다.
- lifecycle이 구조로 박혀 있어 “준비/정리/증거 저장”을 반복 없이 일관되게 만들 수 있다.
- SLI/SLO를 Go 코드로 직접 정의할 수 있어, custom 측정/판정(Go 로직)을 CAT에 그대로 반영할 수 있다.
- Eventually/Consistently로 “복구/안정화 같은 시간 기반 조건”을 자연스럽게 모델링한다.

## 2) 실전에서의 단점
- k6처럼 “요청 지표 집계(p95 등)”를 기본 제공하지 않는다. SLI 집계는 Go 코드로 직접 구현해야 한다.
- 대규모 부하/트래픽 생성에는 적합하지 않다(HTTP 수준도 결국 코드/러너 제약을 받는다).
- 결과 파일 스키마를 CAT 표준으로 정규화하려면 adapter/유사 래퍼가 필요할 수 있다(프레임워크만으로 해결되지 않음).

## 3) k6와의 본질적 차이
- k6: “부하 발생 + metric 집계 + threshold 판정”이 도구 코어에 내장되어 있다. 외부 관측 기반 CAT에서 강하다.
- Ginkgo: “측정과 단언이 전부 테스트 코드에 있다”. 즉, custom logic(특정 상태 머신/복구 정의/합성 판정)에 강하다.

따라서 Ginkgo는 “내가 정의한 의미 그대로의 수용 판정”을 만들기 위한 엔진 후보이고,
k6는 “외부 관측 SLI를 빠르게 만들기 위한 실행/게이팅 도구”에 더 가깝다.

## 4) Ginkgo가 특히 잘 맞는 CAT 시나리오
- Conformance에 가까운 “상태/이벤트 기반 합격 조건”
- recovery, stabilization, readiness처럼 “시간 기반 조건”이 핵심인 시나리오
- 여러 관측을 조합해서 최종 PASS/FAIL을 정의해야 하는 경우(Go 코드로 합성 판정)
- 복잡한 workflow(다단계 step, 상태 전환)를 테스트 코드로 직접 조립해야 하는 경우

## 5) Ginkgo가 불리한 경우
- 수용 기준이 “대규모 트래픽 지표(p95/p99, high RPS)” 중심이고, 도구 기본 집계를 바로 쓰고 싶을 때
- 도메인 특화 수학/통계가 많아 SLI 계산 코드가 커지는 경우(유지보수 부담)
- parallel execution이 필요한데 공유 자원을 엄격히 제어하기 어려운 경우(간헐 실패 위험)
