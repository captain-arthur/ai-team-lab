# 05-Review: Ginkgo minimal custom CAT Job 평가

## 구현 난이도
- Go + Ginkgo/Gomega 조합으로 **단일 테스트**를 구성하는 데 필요한 코드가 많지 않았다.
- 핵심은
  - SLI 계산 로직을 테스트 코드 내부에 넣는 것
  - `DeferCleanup`으로 결과 파일 생성을 보장하는 것
  - `Expect(...).To(Equal("PASS"))`로 PASS/FAIL 단언을 명확히 하는 것이다.

## k6 대비 장점(이 구현 기준 체감)
- “완전히 커스텀한 측정/판정 로직”을 Go 코드로 직접 통제할 수 있다.
- 결과 파일 저장까지 테스트 코드 내부에서 끝낼 수 있어, adapter/파서가 단순해질 수 있다(표준화는 별도 과제).

## k6 대비 단점(제약)
- k6처럼 기본 제공 metric/집계 모델이 아니라, SLI 집계를 우리가 직접 구현해야 한다.
- 시나리오를 외부 트래픽 생성기로 만들 경우(예: ingress 부하), 네트워크/부하 생성/타이밍 제어를 추가로 구현해야 한다.

## Ginkgo가 CAT Job으로 적합한지(체감 기준 결론)
- **적합**: “측정과 단언이 Go 코드로 정의 가능한 CAT”에 대해, 완전 커스텀이 필요한 경우.
- **제한**: “트래픽 중심의 부하/지표 집계”를 기본 모델로 제공받고 싶다면, k6가 더 빠르게 맞을 수 있다.
