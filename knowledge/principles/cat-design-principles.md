# CAT Design Principles (CAT 설계 원칙)

Cluster Acceptance Test 시스템을 설계·구현할 때 따를 원칙이다. 플랫폼/SRE 관점에서 실용적이고 유지보수 가능한 방향을 담았다.

---

## 도구에 대한 태도

- **완벽한 단일 도구는 없다.** 한 도구만으로 conformance, 기능성, 성능, 부하, 팀 정책을 모두 만족시키기는 어렵다.
- **여러 도구를 실용적으로 조합한다.** Conformance는 Sonobuoy, 부하는 ClusterLoader2, 팀 체크는 스크립트/KUTTL 등 역할별로 적합한 도구를 쓰고, 하나의 “CAT run”으로 묶는다.
- **특정 도구에 대한 의존도를 최소화한다.** 도구 A가 사라지거나 바뀌어도, 테스트 시나리오·SLO 정의·결과 형식은 유지할 수 있도록 인터페이스(실행 계약, 디렉터리 규약)를 둔다.
- **도구는 가급적 as-is로 사용한다.** 내부 소스 코드를 수정하지 않고, 설정·플러그인·래퍼로 연동한다. 포크·패치 유지보수 부담을 피한다.

---

## 테스트의 단위: Job

- 하나의 **테스트**는 하나의 **Job**으로 본다.
- Job은 다음 세 요소로 구성된다:

| 요소 | 역할 |
|------|------|
| **Scenario injector** | 부하·트래픽·시나리오를 클러스터에 주입한다 (예: ClusterLoader2 시나리오, 리소스 생성). |
| **SLI/SLO measurement** | 정의된 SLI를 측정하고, SLO 준수 여부를 판단한다. |
| **Assertion** | 측정 결과·상태를 검사해 pass/fail을 낸다. |

- Job은 “시나리오 주입 → 측정 → 단언” 순서로 실행되며, 결과는 **정의된 디렉터리**에 저장한다.

---

## 결과 저장 규약

- **각 Job은 결과를 정의된 디렉터리에 저장한다.** 형식·경로를 팀 규약으로 통일하면, 수집·리포팅·시각화가 쉬워진다.
- 일관된 형식(예: JUnit, JSON, 공통 스키마)으로 저장하면, **자동 시각화**가 가능해진다. “결과만 쌓이면 대시보드/리포트가 나온다”를 목표로 설계한다.

---

## 오케스트레이션

- **단순한 오케스트레이션이 좋다.** Prow 같은 무겁고 복잡한 시스템은 “명확히 필요할 때만” 도입한다.
- 일반적으로는 **순차 실행 스크립트 + runbook** 또는 **가벼운 Job runner**로 충분하다. CAT run 하나를 “Sonobuoy → 부하 테스트 Job → 커스텀 Job → 수집 → 리포팅” 순서로 묶는 수준을 유지한다.

---

## 현실적인 출발점

- **당장 가장 현실적인 시작**은 **ClusterLoader2 부하 테스트**를 활용해, 하나 이상의 **명확한 수용 테스트 케이스**를 만드는 것이다.
- ClusterLoader2로 시나리오를 주입하고, 그 위에 SLI/SLO 측정·단언을 정의해 “이 시나리오에서 이 SLO를 만족하면 수용”이라는 Job을 완성한다.
- 그 다음 Conformance(Sonobuoy), 다른 부하 시나리오, 팀 정책 테스트를 같은 “Job + 결과 디렉터리” 규약으로 확장한다.

---

## 요약

- 단일 도구 의존 배제, 다중 도구 실용적 조합, as-is 사용.
- 테스트 = Job (scenario injector + SLI/SLO measurement + assertion).
- Job 결과는 정의된 디렉터리·일관된 형식으로 저장 → 자동 시각화 가능.
- 오케스트레이션은 단순하게; Prow급은 필요할 때만.
- 출발점: ClusterLoader2 기반 수용 테스트 케이스부터 구체화.
